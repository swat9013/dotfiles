# ツール比較リファレンス

## 評価軸別比較

| 評価軸 | pandas | Polars | Dask | DuckDB | scikit-learn |
|--------|--------|--------|------|--------|--------------|
| ~1GB性能 | ++ | + | - | ++ | + |
| 1-10GB性能 | - | ++ | + | ++ | -- |
| 10GB+性能 | -- | ++ | ++ | + | -- |
| エコシステム | ++ | - | + | - | ++ |
| 学習コスト | ++ | - | - | + | + |
| メモリ効率 | - | ++ | + | ++ | - |
| 遅延評価 | -- | ++ | ++ | ++ | -- |
| 運用コスト | ++ | ++ | -- | ++ | ++ |

## 選ぶ理由 / 選ばない理由

| ツール | 選ぶ理由 | 選ばない理由 |
|--------|----------|--------------|
| **pandas** | エコシステム最高、学習リソース豊富、sklearn/matplotlibと統合済 | 10GB+でOOM、遅延評価なし、大規模では致命的 |
| **Polars** | pandas比5-30倍高速、メモリ効率、遅延評価で大規模対応 | エコシステム不完全（numpy前提ライブラリ非対応）、API変動リスク |
| **Dask** | TB規模の分散処理、pandas互換APIで移行コスト低 | 単機ではPolars/DuckDBより遅い、クラスタ運用コスト大 |
| **DuckDB** | SQL直感的、自動クエリ最適化、OOM回避、ファイル直接クエリ | 非構造化ロジック（条件分岐多い変換）が困難、分散非対応 |
| **narwhals** | マルチバックエンド統一API（ライブラリ作者向け） | ニッチ用途、抽象化オーバーヘッド存在 |

## 除外ツールと理由

**Vaex**
- 開発ペースが低下傾向で将来性に懸念
- メモリマッピング型で探索分析向きだが、同用途をDuckDB/Polarsで代替可能
- Polarsが上位互換と判断

**Modin**
- 0.x系でAPI安定性に不安
- DaskまたはRayのラッパーに過ぎず、直接Daskを使う方が挙動を制御しやすい
- 抽象化のデメリット（デバッグ困難）がメリット（drop-in置換）を上回る

## 組み合わせパターン

| パターン | 用途 | 補足 |
|----------|------|------|
| pandas + scikit-learn Pipeline | 小〜中規模ML標準構成 | Pipeline内でColumnTransformerと組み合わせ |
| DuckDB + Polars | 中〜大規模ETL | DuckDBでSQL変換→Polarsで高速後処理、相互補完 |
| Polars → pandas変換 | エコシステム連携 | `.to_pandas()`で変換、viz/sklearn連携時のみ |

## 長期的展望

- **pandas 3.0**: Copy-on-Write（CoW）+ PyArrowバックエンドで性能改善傾向。既存コードの互換性は維持
- **Polars**: 1-2年で成熟見込み。APIが安定化すれば中規模以上の第一選択肢になる
- **DuckDB**: データエンジニアリングでの採用加速。SQLベース前処理の標準候補
