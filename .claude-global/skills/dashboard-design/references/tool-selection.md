## 1. 意思決定フロー詳細

SKILL.md §5 の3問分岐を補足する判断基準。

### Q1: データソースの性質

| 性質 | 判定基準 | 行き先 |
|------|---------|--------|
| 時系列メトリクス | Prometheus/InfluxDB/Loki 等の監視スタックが既にある | → Grafana（他を検討する理由がない） |
| SQL DB | PostgreSQL/BigQuery/Snowflake 等にデータがある | → Q2 |
| ファイル/API/Python処理 | CSV/JSON/API応答、またはPandasで前処理した結果 | → Q3 |
| 混在 | 時系列+SQL等、複数性質が混在 | → 主たるデータソースで判断。補助データソースは選定後にプラグイン/コネクタで対応 |

### Q2: 組織的BI基盤の必要性

「組織的BI基盤」= 複数チームが共有し、権限管理・埋め込み・アドホックSQL実行が必要な環境。

| 条件 | 判定 |
|------|------|
| SQL知識のある分析者が複数いる + インフラ運用チームがある | → Superset（またはPreset Cloud） |
| 上記を満たすがインフラ運用を避けたい | → Preset Cloud（Supersetのマネージド版） |
| 個人〜小チームで共有する程度 | → Streamlit（構築速度優先） |

### Q3: デプロイ制約

| 制約 | 判定 |
|------|------|
| サーバー運用不可、静的ファイルのみ | → Plotly.js + HTML |
| サーバー可、Python完結で最速構築 | → Streamlit |
| サーバー可、Git統合・再現性・チームレビュー重視 | → marimo |
| プロトタイプ段階、ライブラリ自由に試したい | → Jupyter + Voilà（本番化時に移行コスト発生を許容） |

## 2. ツールカテゴリ別の判断ポイント

各ツールの詳細機能は公式ドキュメントを参照。ここでは**選定時に見落としやすい判断ポイント**のみ記載。

### Grafana
- **強み**: 時系列DBとの統合が圧倒的。アラート→通知→エスカレーションが一気通貫
- **見落とし**: ビジネスKPIダッシュボードには不向き。SQLデータの複雑な結合・集計は苦手
- **ライセンス注意**: AGPL v3。組織のOSSポリシーを事前確認

### Apache Superset
- **強み**: SQL中心のBI。ノーコードビルダー + SQL Lab の二面性で技術/非技術者が共存
- **見落とし**: 本番運用にはPostgreSQL + Redis + Celery + ロードバランサーが必須。「Docker Composeで試せた」から「本番運用」までの距離が大きい
- **コスト判断**: self-hosted の運用コスト > Preset Cloud の SaaS 費用になるケースが多い

### Streamlit
- **強み**: Python→ダッシュボードの最短距離。学習コスト最小
- **見落とし**: 全体再実行モデル（ウィジェット変更で毎回全スクリプト実行）。キャッシング設計が必須。テーマ2色・レイアウト垂直フロー基本でデザイン自由度は低い
- **方向性**: Snowflake統合（SiS）へのフォーカスが増加中。OSSは継続だが注視

### marimo
- **強み**: リアクティブ実行でJupyterの隠れた状態問題を解決。Pure Python形式でGit diff/PRが自然
- **見落とし**: エコシステム未成熟（Jupyter比20-30%）。WASMデプロイは2GBメモリ上限あり
- **成長性**: v1.0ロードマップ進行中。急成長しているが、本番クリティカル用途には成熟度を確認

### Plotly.js + HTML
- **強み**: 運用コスト完全ゼロ。GitHub Pages/S3で配信するだけ
- **見落とし**: クロスフィルタリング（グラフ間連動）は非ネイティブで自前JS実装が必要。DB直結不可（データはビルド時に静的ファイル化するか、fetch APIで取得）
- **次のステップ**: サーバーサイド処理が必要になったらPlotly Dash（Python）への移行パスがある

### Jupyter + Voilà
- **強み**: ライブラリ選択の自由度が最大。探索→可視化の試行錯誤に最適
- **見落とし**: .ipynbはGit差分に不向き。セル実行順序依存で再現性が脆い。ダッシュボードとしての本番運用にはVoilàサーバーのカーネル管理が必要
- **位置づけ**: プロトタイプ・検証段階に最適。本番ダッシュボードにはStreamlit/marimoへの移行を推奨

## 3. トレードオフマトリクス

| Xを優先すると | Yを犠牲にする | 具体例 |
|-------------|-------------|--------|
| セットアップ速度 | カスタマイズ自由度 | Streamlit: 即起動だがテーマ・レイアウト制約 |
| サーバーレス運用 | リアルタイムデータ接続 | Plotly.js+HTML: 運用不要だがDB直結不可 |
| リアクティブ再現性 | 実行順序の自由度 | marimo: 再現性保証だがJupyter的な試行錯誤は制約 |
| 組織的BI機能 | インフラ運用負荷 | Superset: 権限管理・埋め込み充実だがRedis/Celery/DB保守必須 |
| 時系列特化 | 汎用BI分析 | Grafana: 監視最適だがビジネスKPIの複雑集計は苦手 |
| Python完結 | フロントエンド制御 | Streamlit/marimo: JS不要だがCSS/DOM単位の細かい制御は困難 |

## 4. ツール選定のアンチパターン

| パターン | 症状 | 根本原因 | 修正方向 |
|---------|------|---------|---------|
| **BI基盤でメトリクス監視** | Supersetで時系列メトリクスを表示。アラート設定が困難 | データソースの性質（Q1）の誤判定 | 時系列 → Grafana、BI → Superset と分離 |
| **過剰インフラ** | 個人ダッシュボードにSuperset本番構成を構築 | 閲覧者規模の過大評価 | 個人〜小チーム → Streamlit/marimo |
| **プロトタイプの本番化** | Jupyter Notebookをそのまま本番ダッシュボードとして運用 | 移行タイミングの見逃し | 検証完了後にStreamlit/marimoに移行 |
| **フルカスタムJS** | Plotly.jsでクロスフィルタ・認証・状態管理を全て自前実装 | ツールの限界を超えた要件拡大 | 要件がフレームワーク級 → Dash/Streamlit/Supersetに移行 |
| **ツール統一への固執** | 全用途を1ツールでカバーしようとする | 「標準化」の目的化 | 監視はGrafana、BIはSuperset等、カテゴリで分ける |

## 5. Gotchas

**「試せた」と「運用できる」は別**
Docker Composeでの動作確認と本番運用の距離を見誤るケースが多い。特にSupersetは試用→本番のギャップが大きい（メタデータDB移行、Redis/Celery構築、認証統合）。

**ツール選定はメトリクス設計の後**
「何を表示するか」が決まる前にツールを選ぶと、ツールの機能に合わせてメトリクスを歪める。SKILL.md §1-4 → §5 の順序を守る。

**静的ダッシュボードの更新設計**
Plotly.js+HTMLは運用コストゼロだが、データ更新はゼロコストではない。CI/CDでの定期ビルド、またはfetch APIでのデータ取得の設計が別途必要。

**マネージドSaaSの検討**
self-hosted を前提にしがちだが、Preset Cloud（Superset）、Grafana Cloud、Streamlit Community Cloud など、マネージド版が存在する。運用チームの有無で判断が変わる。

**選定後の調査ポイント**
ツール候補を1-2に絞った後、実装前に以下を公式ドキュメントで確認する:
- 最新バージョンと破壊的変更の有無
- 既存データソースとの接続方法
- デプロイ手順と運用要件
- ライセンス（AGPL/Apache/MIT）と組織ポリシーの整合
