# アンチパターン診断手順と改善パス

SKILL.md の早見表を展開した各アンチパターンの診断ステップと改善パス。

## Metric Overload（メトリクス過多）

**診断手順**:
1. 表示メトリクス数を数える（9 個超 = 警戒、15 個超 = 要対応）
2. 全メトリクスに Decision Test を適用する
3. Q1 に即答できないメトリクスをリストアップする

**改善パス**:
- Q1 に即答できないものを「支援指標」に降格 → Layer 2 or Layer 3 に移動
- Q2 にも答えられないものを削除候補としてマーク
- 削除前に「30日間非表示にして誰かが気づくか」の猶予期間を設ける

**Gotcha**: 「減らすと情報が失われる」という反論は、「今実際に使われているか」の利用ログで反証する。

## 文脈なし数値

**診断手順**:
1. 各メトリクスに「目標値・前期間比・トレンド」のうち何個あるかを確認
2. 0個 = 要対応、1個 = 改善余地あり、2個以上 = OK

**改善パス**:
- 最低限「目標値との比（達成率 %）」を追加する（最小コスト）
- 前期間比を追加することで「方向と速度」が見える
- スパークライン追加で「短期変動 vs 長期傾向」が区別できる

## Vanity優先配置

**診断手順**:
1. Overview 左上または最上部のメトリクスを特定する
2. そのメトリクスに 3A 評価を適用する
3. Actionable が「要対応」なら Vanity 配置と判定する

**改善パス**:
- 3A 評価で Actionable な指標を Overview の上位に再配置する
- 単調増加する累計数値（総ユーザー数等）は Layer 2 に移動する
- 「見栄えが良い数字」を経営報告用に別ダッシュボードへ分離することで両立を図る

## 診断能力欠落

**診断手順**:
1. 異常が見えたとき（例：売上20%低下）、ダッシュボード内でその原因を追跡できるか確認
2. 「追跡できない → 別ツールに移動が必要」= 要対応

**改善パス**:
- Layer 3 に根本原因分析パス（ドリルダウン）を設ける
- ドリルダウンは「異常を見た人が自律して原因を特定できる」ことを目標とする
- 完全な分析機能は別の分析ツールに委ねる。ダッシュボードは「どこを掘るかを示す」役割に留める

**判断基準**: ダッシュボードで「なぜ」まで答えようとすると分析ツール化する。「問題の存在と規模を即座に把握し、次の掘り下げ先を示す」が適切なスコープ。

## 更新停止（ゴーストダッシュボード）

**診断手順**:
1. データの最終更新日を確認する
2. 所有者が現在も在籍・担当しているか確認する
3. 過去30日の閲覧回数を確認する

**改善パス**:
- 所有者を明示的に再アサインする（担当者名+連絡先をダッシュボード内に表示）
- 四半期レビュー（ASSESS→PLAN→ACT→REVIEW）を所有者のカレンダーに登録する
- 「90日間更新なし + 30日間閲覧なし」を廃止トリガーとする

## Gotchas

**診断は「評価」ではなく「改善のための対話」**

診断でスコアや合否判定を前面に出すと、ダッシュボード所有者は防衛的になり改善議論が進まない。各段階の出力は常に「次に何をするか」の形式で提示する。

| NG | OK |
|-----|-----|
| 「このダッシュボードは3A基準で60点です」 | 「Actionable の観点で、X メトリクスのアクション導線が不明確でした。Q1 に答えられる形に整理しましょう」 |
| 「Vanity metric が多すぎます」 | 「Overview の上位3件のうち2件が累計数値でした。Retention Rate を上位に移動すると意思決定に使いやすくなります」 |

**削除は最後の手段**: 「30日間非表示」→「誰も気づかなければ削除」の段階を踏む。

**利用ログがない場合の罠**: Decision Test の Q3（所有者）に答えられるかを代理指標として使う。

**Leading/Lagging の境界は文脈依存**: 分類より「このメトリクスが変化したら、次に何が起きるか」を問うことが本質。

**診断の起点を誤ると全体が狂う**: Stage 1 で計算ロジックの確認先（Auditable）もセットで記録する。
