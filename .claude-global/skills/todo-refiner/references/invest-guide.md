# INVEST原則ガイド

タスクを「AIに渡して実行可能」な品質にするための基準。

## INVEST原則

| 原則 | 意味 | チェック観点 |
|-----|------|------------|
| **I**ndependent | 独立している | 他タスク完了を待たずに着手可能か |
| **N**egotiable | 交渉可能 | 実装方法に柔軟性があるか |
| **V**aluable | 価値がある | なぜこのタスクが必要か説明できるか |
| **E**stimable | 見積もり可能 | 作業量が想像できるか |
| **S**mall | 小さい | 1セッションで完了できるか |
| **T**estable | テスト可能 | 完了条件が明確か |

## 本スキルでの重点項目

`tt`コマンドでの利用を想定し、以下を重視:

### Valuable（価値）

タスクの「なぜ」を明示。AIが優先度や重要性を判断できる。

```
悪い例: ターミナルタイトルを変える
良い例: ターミナルタイトルをカレントディレクトリ名に変更する。価値: 複数タブの識別が容易に
```

### Testable（完了条件）

何をもって完了とするかを明示。AIが作業終了を判断できる。

```
悪い例: 設定を改善する
良い例: 〜する。完了: ghostty設定変更済み、タイトルが正しく表示される
```

## 曖昧タスクの判定基準

以下に該当する場合は「曖昧」と判定し、質問で具体化:

1. **What不明**: 何をするか具体的でない（例: 「改善する」「調整する」）
2. **対象不明**: 何に対する作業か不明（例: 「長時間実行」→何の？）
3. **ゴール不明**: 完了状態が想像できない（例: 「育てる」）

## 方針タスクの変換

「〜を使う」「〜に気をつける」のような方針は、設定タスクに変換:

```
Before: TDDを要所で使う
After:  CLAUDE.mdにTDDルールを追加する（バグ修正時はテスト先行）。価値: AIがTDDを適切に使用。完了: rules/に記載済み
```
