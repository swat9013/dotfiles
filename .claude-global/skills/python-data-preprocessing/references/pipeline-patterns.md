# pipeline-patterns.md

## 1. 欠損値処理の判断フロー

```
欠損パターン分析（missingno可視化）
  ↓
MCAR/MAR/MNAR判定
  ├─ MCAR（ランダム欠損）→ データ量十分なら削除可。SimpleImputerで平均値/中央値補完
  ├─ MAR（他変数依存） → KNNImputer、IterativeImputer(MICE)で多重補完
  └─ MNAR（欠損自体が情報）→ 欠損フラグ列追加 + 中央値補完。領域専門家に相談
```

## 2. メモリ最適化

### dtype別削減率

| 手法 | 効果 | 適用条件 |
|------|------|----------|
| int64→int8 | 最大87.5%削減 | 値域が限定的な整数列 |
| float64→float32 | 50%削減 | 高精度が不要な浮動小数点列 |
| object→category | 90%+削減 | 低カーディナリティ(〜50種類)の文字列列 |
| chunksize指定 | メモリ上限内で任意サイズ処理 | GB級CSVの読み込み |
| PyArrow文字列型 | 文字列メモリ効率改善 | pandas 3.0ではデフォルト |
| usecols | 不要列を除外 | 列数が多いデータセット |

### OOM対策フロー

```
OOM発生時の対応
├─ Step1: dtype最適化 + usecols → 不要な型拡張・列を除去
├─ Step2: chunksize指定 → メモリ上限内で分割処理
├─ Step3: ツール変更 → Polars/DuckDBへ移行（遅延評価でメモリ効率化）
└─ Step4: 分散処理 → Dask（TB規模、クラスタ環境あり）
```

## 3. データ検証 2層構成

| レイヤー | ツール | 検証対象 | 用途 |
|----------|--------|----------|------|
| 行レベル | Pydantic | 個別レコードの型・値域 | API入力、ETLの行単位検証 |
| テーブルレベル | Pandera | DataFrame全体のスキーマ・分布 | パイプライン中間検証、品質ゲート |

本番環境: Great Expectations — 自動アラート・マルチエンジン対応・Data Docs生成

## 4. 再現性確保チェックリスト

- [ ] シード一元管理: random/numpy/torch/CUDAを統一 `set_seed()` 関数で設定
- [ ] random_state明示: train_test_split、KFold、全推定器に整数を渡す
- [ ] パラメータ保存: 前処理パラメータ + Gitコミットハッシュ + ライブラリバージョンを記録
- [ ] 決定的データロード: DBクエリに `ORDER BY` 付与、サンプリングにシード指定

## 5. テスト戦略

| テスト種別 | 内容 | ツール |
|------------|------|--------|
| ユニットテスト | 欠損値処理・スケーリング一貫性・エンコーディング確定性 | pytest |
| スキーマテスト | DataFrame形状・型・値域の自動検証 | Pandera |
| 再現性テスト | 同一シードで2回実行し結果一致確認 | pytest |

カバレッジ基準: 新規コード80%以上、クリティカルパス95%以上

## 6. パフォーマンス計測

- ボトルネック特定: `cProfile` / `memory_profiler` でフェーズ別に時間・メモリを記録
- 処理速度改善: `apply()` → ベクトル化操作、条件分岐 → `np.select()`、文字列 → `.str` アクセサ
