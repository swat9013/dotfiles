# Problem Domain Modeling

**このモデリングセッションはユーザーが終了を宣言するまで、反復ループを継続する。**

## 動作原理

- **使用文脈での採取**: 語彙は定義でなく、ユーザーが使う文脈と共に記録する。定義を先に与えるとユーザーの認知が歪む
- **差異関係での意味化**: 語彙の意味は他語彙との差異関係で定まる。独立した語の定義に拘らず、関係ネットワークで意味を浮かび上がらせる

## 起動手順

1. **モデルファイルを作成する**
   - パス: `.claude/domain-models/YYYY-MM-DD-HHMMSS-{topic}.md`
   - `{topic}` は `$ARGUMENTS` から英数字+ハイフン30文字以内で生成
   - 初期内容は `references/output-schema.md` のテンプレートに従う（`status: open`, `loop_count: 0`）

2. **資料の有無を判定する**
   - `$ARGUMENTS` にファイルパス（`.md`/`.txt`/`.pdf` 等の拡張子 or 存在するパス）または URL（`http://`/`https://`）を検出 → **資料先行モード**
   - そうでなければ → **通常モード**

3. **セッション開始を宣言し、最初の語彙採取に入る**
   - `$ARGUMENTS` から読み取れる課題の概要を自分の言葉で簡潔に要約する
   - 認識の正しさは確認せず、まず語彙採取に進む（合意形成は採取後のモデル化フェーズ以降で行う）

### 資料先行モード

1. 資料を取得する（ファイル → `Read`、URL → `WebFetch`）
2. 固有名詞・頻出用語・専門用語を語彙候補として抽出する
   - **上限20語**、広く浅く採取する（深掘りは歪み検証・次ループで行う）
   - 資料が大きい場合は章立て・見出しから代表語を拾う（全文精読はしない）
3. 抽出した語彙を `## ループ1 — 語彙採取` に書き込む
4. ユーザーに提示: 「資料から採取した語彙です。追加・修正したい語彙はありますか？」
5. 次フェーズ（モデル化）へ進む

**注意**: 資料の語彙は公式・文書化されたものに偏る。暗黙語彙（俗称・ニックネーム）はループ2以降で別途採取する。

### 通常モード

1. ユーザーに質問: 「この課題で日常的に使っている言葉を3〜5個挙げてください」
2. 専門用語・略語・チーム固有の言い回しも歓迎することを伝える
3. 応答を `## ループ1 — 語彙採取` に書き込む

## 反復ループ

各ループは以下の4フェーズで構成される。ループNはセッション開始から数えた回数。

### ループN: 語彙採取

- **目的**: 一次源から語彙を浅く広く収集する
- **方法**: ユーザーが自然に使う言葉をそのまま記録する。定義や解釈は加えない
- **出力**: モデルファイルの `## ループN — 語彙採取` に書き込む
- **ループ2以降の暗黙語彙採取**: 「資料・公式ドキュメントに書かれていない、日常で使う言葉はありますか？」を追加で問う

### ループN: モデル化

- **目的**: 採取した語彙を関係ネットワークとして整理する
- **方法**: 以下の関係パターンで語彙間のエッジを抽出する
  - 「AはBを含む」「CはDを引き起こす」「EとFは競合する」「GはHに依存する」「IはJを前提とする」
  - 差異関係が曖昧な場合はエッジを引かず、歪みとして次フェーズへ送る
- **出力**:
  1. 関係ネットワーク（エッジリスト形式）を `## ループN — モデル（関係ネットワーク）` に書き込む
  2. drawio XML を生成し `.claude/domain-models/YYYY-MM-DD-HHMMSS-{topic}-loopN.drawio` に保存（ループごとに完全再生成、過去ループの drawio は歴史として保持）
  3. PNG をエクスポート: `/Applications/draw.io.app/Contents/MacOS/draw.io -x -f png -b 10 -o <name>.drawio.png <name>.drawio`
  4. `## ループN — モデル図` セクションに画像と編集用ファイルへのリンクを記載
- **ユーザー確認は行わない**: モデル化段階で確認を挟むと自己解決プロトコル（訂正フェーズ参照）を無効化する。認識合わせは訂正フェーズの `user` 歪み停止時に PNG リンクと共に行う

**drawio 生成の詳細**: drawio XML の構造・CLI フラグは `drawio` スキル（`.claude-global/skills/drawio/SKILL.md`）を参照。`-e` フラグは破損原因のため禁止。

### ループN: 歪み検証

- **目的**: モデルの歪み・欠落・矛盾・命名リスク・標準語彙差分を発見する
- **方法**: モデル化フェーズで生成した **drawio 図（PNG）を参照しながら** 以下の観点で点検する
  - PNG 上で循環・孤立ノード・過密領域を目視確認する（エッジリストだけでは見落としやすい構造的歪み）
  - 5種類の歪み種別を順に点検する

**5種類の歪み種別** (`references/output-schema.md` 参照):

| 種別 | 検出観点 | 訂正責務（判別例） |
|------|---------|------------------|
| MISSING | 関係の片側が不明、原因候補が未分類、重要な語彙の欠落 | `auto`: 既採取語彙のノード配置漏れ / `user`: 「多忙 vs 意図的無視」など意味レベルの原因候補 |
| CONFLICT | 相反する関係が並立、矛盾する前提 | `auto`: 同一エッジが「含む」と「依存する」で二重定義 / `user`: 「非同期推奨」と「即レス文化」など運用文化の矛盾 |
| AMBIGUOUS | 因果方向が不明、語彙の指す範囲が不定 | 原則 `user`: 因果方向確定は実運用経験が必要（例外: A→B と B→A 両エッジが既存なら `auto` で統合） |
| STANDARD-GAP | 業界標準語彙（DDD、分野用語など）との差分。**差分確認のみ、補正はしない** | 常に `auto`: 差分を記録するのみで命名変更は行わない |
| NAMING-RISK | 陳腐化する命名（採用技術名・製品ブランド・役職・人名・解決策混入・略語） | `auto`: 6パターン（output-schema.md 参照）への該当検出・抽象化候補提示 / `user`: 最終命名の選定 |

- **出力**: `## ループN — 歪み記録` に `- [歪み種別] 説明 (auto|user)` 形式で記録（末尾の `auto`/`user` で訂正責務を明示）

### ループN: 訂正と引き継ぎ

- **目的**: 歪み検証の結果をモデルに反映し、次ループへの足がかりを作る
- **自己解決プロトコル（フロー確定版）**:

  **Step 1: auto訂正（現ループ内で完結）**
  - `auto` 区分の歪みを現ループ内で自律訂正する
  - 訂正結果は `## ループN — 次ループへの引き継ぎ` に「auto訂正サマリ」として記録する
  - この時点では `## ループN+1` セクションは**まだ作成しない**（モデル本体への反映は後続ステップで行う）

  **Step 2: user歪み判定による分岐**
  - **`user` 区分がゼロ**: `loop_count` をインクリメントし、そのまま次ループの語彙採取・モデル化フェーズへ進む（連続進行）
  - **`user` 区分が1件以上**: ここで停止し、以下の2パート構造でユーザーに返す:
    1. **自動で解消した結果**: `auto` 歪みの訂正内容サマリ（Step 1 の内容）
    2. **あなたの判断が必要**: `user` 歪みを具体的な選択肢（A/B/C）として提示。各選択肢の根拠も添える
  - **停止中の状態**: `loop_count` は据え置き、モデル本体（`## ループN+1` セクション）は未作成のまま

  **Step 3: ユーザー回答後の反映（user停止時のみ）**
  - ユーザー回答を受け取ったら `loop_count` をインクリメント
  - auto訂正結果 + user回答を統合して `## ループN+1 — 語彙採取` / `モデル（関係ネットワーク）` / `モデル図` を生成
  - 次ループの歪み検証フェーズへ進む

- **ユーザー判断が必要な典型ポイント**:
  - 運用依存の閾値決定（定量化基準、粒度選択）
  - 機能的に等価な命名候補間の主観的選好
  - 実運用経験に依存する検証（因果方向の確定、実例の有無）
  - 設計所有範囲や責任分掌の最終判断
- **不変条件**:
  - 現ループのセクションは追記せず歴史として保持する（過去ループの .drawio も保持）
  - user停止中は `## ループN+1` セクションを書かない（中途半端な状態をファイルに残さない）
- **出力**:
  - 引き継ぎ事項 → 現ループの `## ループN — 次ループへの引き継ぎ` に記録（auto訂正サマリ + `user` 歪みが残っていればそれも明示）

## 停止条件

ユーザーが以下のいずれかを宣言したとき、ループを終了する:

- 「終了」「閉じる」「close」「モデルを確定する」「このループで十分」等

**終了時の処理:**

1. モデルファイルの `status: open` を `status: closed` に更新する
2. モデルファイルの末尾に `## セッションサマリ` セクションを追記する
   - 未解決の歪み内訳は、モデルファイル内の `[MISSING]` `[CONFLICT]` `[AMBIGUOUS]` `[STANDARD-GAP]` `[NAMING-RISK]` の出現数をカウントして記載する
   - 書式は `references/output-schema.md` 参照
3. 成果物パス（モデル .md・最終ループの drawio/PNG）をユーザーに案内する

自律的停止（飽和・収束による）は実装しない。ユーザーの宣言のみが停止条件。
