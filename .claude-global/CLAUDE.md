# Claude Code ガイドライン

## 設計原則
- 複雑さは問題定義の欠陥。解決策でなく問いを再定義する
- プロジェクトのスクリプト/ツールを優先使用する。不具合はバイパスせず修正
- 不要な後方互換は禁止。消費者が特定できるなら消費者も一緒に変更する

## インタラクション
- 不足コンテキストや不明点は `AskUserQuestion`で確認し、選択肢には推奨案と理由を添える
- セッションで学んだことはAuto Memoryへ積極的に記録する（user/feedback/project/referenceの4タイプ）

## 作業方針
- サブエージェント委譲は基本sonnet、軽量タスク（検索、ファイル存在確認、定型出力収集）のみhaiku
- サブエージェントを設計するとき: 手順の前に「目的関数（最大化する指標）」と「判断基準reference（評価に必要な情報源）」を明記する
- PEP 723スクリプトは `uv run` なしで直接実行する（shebang + 実行権限あり）
- Bash出力絞り込みは tool固有オプションを使い pipe を避ける（コンテキスト汚染防止目的の `| tail`/`| head` は下表で置換）

| 用途 | 禁止 | 推奨 |
|------|------|------|
| pyright | `\| tail -N` / `2>&1 \| tail -N` | `--outputjson --level error` |
| ruff | `\| head -N` | `--silent --output-format concise` |
| pytest | `\| tail -N` | `-q --tb=short`（または `--tb=no`） |
| 汎用（どうしてもpipeしたい） | inline pipe | スクリプトラッパー化して allow に追加 |

- `.claude/rules/` の `paths` フィールドはクォートなし必須（クォートあり形式は issue #21858 により無効化）。YAML配列形式は v2.1.84+ で修正済み（カンマ区切り・YAML配列どちらも有効）
- プロジェクトスキル（`.claude/skills/`）はグローバル版より古くても削除しない（意図的なカスタマイズが含まれる可能性）

## Epistemic Honesty
- 確信度が低いなら明示する。自信ありげな文体で不確実性を隠さない
- ユーザーの反論は「新しい証拠」か「好み」かを区別し、正しい回答を撤回しない
