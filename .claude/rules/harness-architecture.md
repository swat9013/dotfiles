## 2. 4原則の詳細展開

### 原則1: Guide First, Sense Second

| 領域 | 適用基準 |
|------|---------|
| コード生成 | スキルの指示・テンプレートで事前に構造を決定。PostToolUse でのドリフト修正は禁止 |
| 環境操作 | guard スクリプトによる事前チェック（PreToolUse）を優先。事後のロールバックは設計失敗のサイン |
| 自己管理 | 診断より設計ガイド（rules/）の整備を優先。診断は「逸脱が起きた後」の用途に限定 |

### 原則2: Computational First

| 領域 | Computational（スクリプト化） | Inferential（AI 判断） |
|------|-----------------------------|-----------------------|
| コード生成 | ファイル存在確認、行数カウント、diff 抽出 | レビュー観点の優先度付け、設計判断の評価 |
| 環境操作 | lint/test 実行、終了コード判定、パス検証 | エラーメッセージの診断、リカバリ方針の決定 |
| 自己管理 | 行数超過検出、パス参照の整合性確認 | スキル分割タイミングの判断、コナセンス重要度評価 |

### 原則3: ライフサイクル配置

| フェーズ | タイミング | 配置する制御の種類 |
|---------|-----------|-----------------|
| タスク開始前 | PreToolUse / スキル冒頭 | 前提条件チェック、禁止操作ガード |
| 実行中 | スキル内ステップ | 品質ゲート通過判定、分岐制御 |
| 実行後 | PostToolUse / スキル末尾 | 成果物検証、次フェーズへの引き継ぎ |
| セッション後 | Stop hook | セッションサマリ記録、一時ファイルクリーンアップ |
| 定期 | Cron / 手動トリガー | Harness 劣化検出、依存関係の陳腐化確認 |

### 原則4: 自己メンテナンス

Harness 自体が劣化する。設計時点で劣化検出と修正の仕組みを組み込む（→ §5 Steering Loop）。

## 5. Steering Loop

Harness 自体の劣化（設定の陳腐化・スキル肥大化・コナセンス破綻）を継続的に検出・修正する。知見フロー・計測データの流れ・retrospective パイプライン分離の詳細は `docs/harness-engineering-domain-model.md` §4.2 を参照。

### 劣化のシグナル

| シグナル | 検出方法 | 対応カテゴリ |
|---------|---------|------------|
| スキルが期待どおりに起動しない | description のトリガー漏れを確認 | スキル設計の見直し |
| フックが無効化されたまま残存 | settings.json とスクリプト実体の整合確認 | 設定管理のコナセンス |
| rules/ ファイルが肥大化 | 行数 200 超のファイルを検出 | skills/ への分離 |
| 同一情報の重複 | references/ と rules/ の並走を確認 | 正規ソースへの集約 |
| PermissionRequest の偏り | scan-metrics.py（claude-config Agent 1）の tool_counts 集計 | permissions.allow / guard の見直し |
| 独立レビュー指摘の頻出 | claude-config Agent 2 で検出 | rule/skill への昇格検討 |

### 修正の責務分担

- **診断**: `claude-config` スキル（自己管理領域）が担う
- **claude-config 自体の校正**: `harness-tuning` スキルが担う（スクリプト/Agentプロンプト/references の不具合診断）
- **スキル設計の修正**: skill-design-patterns の設計パターン（claude-config/references/skill-design-patterns.md）に基づき対応。スキル作成の実行は skill-creator が担う
- **フック・settings.json の修正**: `config-management.md` のコナセンス・マップに従う
- **本ドキュメント自体の修正**: Harness 設計の境界・原則が変わったときのみ実施
