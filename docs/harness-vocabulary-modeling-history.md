### A. 上位目的・価値に関する語彙

- **価値観の遵守** [DM §1]: harnessの存在意義。少ない指示でユーザーの設計・実装の価値観に沿った行動をエージェントが自律的に取れること
- **再現性** [DM §1]: 価値観の遵守の前提。別セッションでの同等品質
- **精度** [DM §1]: 価値観の遵守の前提。ハレーション・逸脱の少なさ
- **責務の価値** [DL Turn 1]: 各コンポーネント（スキル/フック/ルール）が最大化すべき価値。ユーザーが「曖昧」と指摘した対象
- **参照点となる価値** [DL Turn 2]: opusレビューや診断が「何を最大化すべきか」を判断するための基準点。これが欠落すると汎用ヒューリスティックに退行する

### B. コンポーネント単位の基準語彙

- **目的関数** [CL, CC, HA §5]: サブエージェント設計時に「最大化する指標」として明記する概念。CLAUDE.mdで「手順の前に目的関数と判断基準referenceを明記する」と義務化。claude-config Agent 1/2/3 に個別設定済み
- **判断基準reference** [CL]: 目的関数とペアで明記する「評価に必要な情報源」。referencesディレクトリ等の参照先を指す
- **最大化する指標** [CL]: 目的関数の言い換え（CLAUDE.md内の定義文中で使用）
- **最大化する価値** [DL Turn 4]: 三部構造化提案での第1要素。定性1文、「これがなければ何がどう壊れるか」
- **達成シグナル** [DL Turn 4]: 三部構造化提案での第2要素。価値が出ている観測可能な兆候、複数並列（Goodhart対策）
- **境界条件** [DL Turn 4, 候補]: 三部構造化提案での第3要素候補。追求時に侵してはならない制約
- **禁則** [DL Turn 4, 候補]: 第3要素の別候補
- **抑止対象** [DL Turn 4, 候補]: 第3要素の別候補

- **success_signals** [DL Turn 2, 却下済み]: 英語混在の初期提案。「達成シグナル」に置換
- **anti_goals** [DL Turn 2, 却下済み]: 英語混在の初期提案。「境界条件」等に置換候補

### C. タスク/フェーズ単位の基準語彙

- **成功基準** [CC, researcher/SKILL.md]: サブエージェント実行が成立したかの判定基準。per-run（1回の実行に閉じる）
- **受け入れ条件** [DC §2]: plan/architect がフェーズごとに定義する条件。breakdown を経て implement が Phase 完了時に検証
- **Phase完了基準** [DC §3]: breakdown テンプレートに転記される条件。受け入れ条件の下流形
- **完了定義** [DM §2.2]: 例として「planの完了定義→LLM判定」と記述。LLM評価による推論的フィードバック
- **高信号フィルタ** [DC §3]: コードレビューで除外基準に該当しない指摘を通すフィルタ。`_shared/review-criteria.md` が正規ソース

### D. 実行時チェック語彙

- **品質ゲート** [DC §3, HA §5]: quality-gate.py のJSON出力 `gate` フィールド。`PASS`/`FAIL`/`SKIP` の3値。Lint/Test/Build の複合結果を集約
- **判定キー** [HA §5, DC §5]: スクリプト出力末尾に配置する機械判定用キー（`GATE:` / `RESULT:`）。スキル側分岐ロジックとの契約
- **変更検出判定キー** [DC §3]: changed-files.sh の `RESULT: NO_CHANGES/CONFIG_ONLY/TOO_LARGE/PROCEED`
- **guardスクリプト** [DM §2.2]: PreToolUse等で動く禁止操作遮断スクリプト（guard-*.sh）
- **permissions.deny** [DM §2.2]: settings.jsonの拒否リスト。guardと二層防御を構成

### E. 劣化・自己メンテナンス語彙

- **劣化シグナル** [HA §5]: Harness自体の劣化を検出する兆候。6種類（スキル非起動、フック残存、行数超過、重複、PermissionRequest偏り、opusレビュー指摘頻出）
- **Steering Loop** [HA §5, DM §4.2]: 劣化検出→診断→修正→検証の運用サイクル
- **逸脱** [DM §1, CC]: 価値観からのズレ。レビュー・診断の主要観測対象
- **捕捉** [DM §4.2]: retrospectiveの責務。知見の収集（memory蓄積）
- **昇格** [DM §4.2]: claude-configの責務。memory/tmp → rule/skill/CLAUDE.md への格上げ判定
- **推論的→計算的の昇格** [DM §2.2]: Feedforward で書かれた規則のうち決定論的なものを Feedback に格上げする判定基準

### F. 制御メカニズム分類語彙

- **Feedforward** [DM §2.2]: 事前方向づけ。rule/skillによる推論的誘導
- **Feedback** [DM §2.2]: 事後検出。hook/deny/opusレビュー/LLM評価
- **推論的** [DM §2.2]: LLMが文意を解釈して反映する制御
- **計算的** [DM §2.2]: スクリプト/設定で決定論的に実行する制御
- **ガード(A)** [DM §2.2]: 計算的制御4分類の1つ。禁止操作の遮断
- **検証(B)** [DM §2.2]: 計算的制御4分類の1つ。品質チェック
- **注入(C)** [DM §2.2]: 計算的制御4分類の1つ。情報制御
- **計測(D)** [DM §2.2]: 計算的制御4分類の1つ。可観測性
- **Guide First, Sense Second** [HA §2, CLAUDE.md]: 事前方向づけ優先の設計原則
- **Computational First** [HA §2]: 計測可能な基準はスクリプト化する設計原則

### G. 計測・データ層語彙

- **計測データ** [DM §4.2, §5]: PermissionRequest、セッション定量、opusレビュー結果の3種
- **定性データ** [DM §4.2]: retrospective collect.py の出力（セッション会話+tmp/review）

### 命名規則の確定（ループ1→ループ2への移行時）

**規則**: 評価・基準系でスコープ依存の語は一律接頭辞化する（例外なし）。

**理由**: LLMは複数ファイル読込で情報がblendされ、ファイル境界が保存されない。例外を設けるとLLM側で境界復元不能。

**適用範囲**: 評価・基準系の語彙（B/C/D/E群）のスコープ依存語のみ。固有名詞（Steering Loop, 3領域）と概念分類語（Feedforward, Computational等）は類似語が存在しないため対象外。

**スコープ階層**: `Harness` > `コンポーネント`（skill/hook/rule単位）> `Phase` > `タスク` > `サブエージェント`

### 主要な帰結

1. **コンポーネント目的関数は Harness の全評価系の参照点**。opusレビュー、claude-config診断、Steering Loop昇格判定、すべてがここを読むべき

4. **品質ゲートは実行時チェック層で、評価基準とは独立**。Phase完了基準の一部を機械的に検証する下位機能

## ループ5 — 語彙採取

**対象歪み**: #2 三部構造の適用範囲、#3 コンポーネントの定義境界

### 5-α. サブエージェント目的関数の粒度（歪み#2）

現状の合意:
- **コンポーネント目的関数** = 三部構造（最大化価値 / 達成シグナル / 境界条件）
- **サブエージェント目的関数** = 1文運用（CLAUDE.md「手順の前に目的関数と判断基準referenceを明記する」現行記述に準拠）

**論点**: サブエージェント目的関数もフル三部構造にすべきか、1文でよいか、中間形にすべきか。

選択肢素案:
- (a) フル三部構造: コンポーネント目的関数と完全対称。重複コスト大、per-runで毎回注入する負荷
- (b) 1文運用のまま: 軽量だが境界条件が落ち、Feedforward自己抑制が効かなくなる懸念
- (c) 中間形: 「1文の最大化価値 + 境界条件1〜2個」。達成シグナルはper-runなのでサブエージェント成功基準に吸収

### 5-β. コンポーネントの定義境界（歪み#3）

現状の合意: skill / hook / rule は明確にコンポーネント。

**論点**: 以下をコンポーネントとみなすか。

- **CLAUDE.md** (global/project): グローバル注入される行動指針
- **memory** (user/feedback/project/reference): 永続化された文脈
- **settings.json**: 権限・フック・MCP設定
- **references/**: スキルが参照する静的知識
- **SKILL.md**: skill本体の一部扱いか、独立コンポーネントか

判定軸候補:
- α. 「目的関数を書けるか」基準（= 最大化価値が定義できるか）
- β. 「Harness劣化シグナルの観測対象になるか」基準
- γ. 「claude-config診断の対象か」基準

### ユーザー判断（採取結果）

**5-α 採用**: **(c) 中間形** — サブエージェント目的関数 = 最大化価値1文 + 境界条件1〜2個。達成シグナルはサブエージェント成功基準に吸収

**5-β 採用**:
- **コンポーネント**: skill / hook / rule / **CLAUDE.md** / **settings.json**
- **非コンポーネント（skill内部詳細）**: references/ / SKILL.md
- **根拠（ユーザー言明）**:
  - CLAUDE.md = 「グローバルのFeedforward」（事前方向づけ層）
  - settings.json = 「Feedback（禁止操作を管理する）」（事後制御層）
  - references/ と SKILL.md = skillコンポーネントの内部詳細。Harnessからはskill単位で観測するが、内部構造としては観測しない

**判定軸（5-βから逆算）**: 「Harnessからの視点で独立に観測されるか」= 粒度軸。α/β/γのいずれでもなく、**観測粒度**が判定軸として機能した

### ループ6での確定事項

#### 1. 用語の2層化

| 階層 | 語 | 含意 |
|------|-----|------|
| 責務軸 | **ドメイン** | 何を達成するか。目的関数を持つ |
| 実装軸 | **コンポーネント** | 何で実装するか。ドメインの実装手段 |

#### 2. ドメイン = harness engineeringの規制カテゴリ3種を採用

参考: https://martinfowler.com/articles/harness-engineering.html

- harness engineering ⊂ context engineering（部分集合）
- context engineeringのWrite/Select/Compress/Isolateは配備手段として下位
- 独自責務軸（段階開示/計算的Feedback/境界提供）は立てない

**3ドメイン**:

##### Maintainability Harness

- **対象**: 生成コード・リポジトリの保守性
- **既存コンポーネント**: coding-principles rule / ruff / pyright / pytest / 品質ゲート
- **最大化価値**: 生成コードが長期保守可能な状態を維持する
- **達成シグナル**:
  - 品質ゲートPASS率が高位で安定
  - 型エラー・lint違反が低位で安定
  - リファクタ容易性が損なわれていない
- **境界条件**:
  - 過剰な自動修正で意図を歪めない
  - 偽陽性で開発を止めない

##### Architecture Fitness Harness

- **対象**: harness自体の健全性（自己メンテナンス）
- **既存コンポーネント**: harness-architecture rule / retrospective / claude-config / 独立レビュー / Steering Loop / コナセンス管理
- **最大化価値**: harness自身の劣化を検出・修正し続ける
- **達成シグナル**:
  - Harness劣化シグナルがSteering Loopで回収される
  - 未解決の逸脱が蓄積しない
  - ドメインモデルと実装の乖離が小さい
- **境界条件**:
  - 自己メンテナンスが本業を侵食しない
  - 計測のための計測にならない

##### Behaviour Harness

- **対象**: エージェント実行時の行動（価値観遵守）
- **既存コンポーネント**: CLAUDE.md / dev-cycle rule / 各workflow skill / settings.json / hook
- **最大化価値**: ユーザー価値観に沿った自律行動を少ない指示で実現
- **達成シグナル**:
  - ユーザー修正指示の反復がない
  - Feedforwardで自己抑制が効く
  - Feedbackで逸脱が検出される
- **境界条件**:
  - 過剰規則化で表現を硬直化させない
  - 常時必要な指示をrulesに退避しない

#### 3. 二軸制御構造（ドメイン横断の下位属性）

- 実行タイプ: Computational / Inferential
- 時系列軸: Feedforward (Guides) / Feedback (Sensors)
- 3ドメイン × 4象限 でコンポーネントを配置可能

#### 4. コンポーネントの複数ドメイン貢献を許容

- 1コンポーネントが複数ドメインに貢献することを認める（当然のこと）
- 例: CLAUDE.mdはBehaviour主、Maintainability補助
- 例: 品質ゲートはMaintainability主、Behaviour補助
- 例: 独立レビューはArchitecture Fitness主、Behaviour補助

#### 5. ループ5成果の再配置

- フラクタル構造 / 委譲価値6類型 / 責務分担4基準 / 静的/動的軸 / 静的定義昇格原則: ドメイン横断で有効なので残す

### 5-γ. フラクタル構造の発見（ユーザー指摘）

**指摘**: 「サブエージェントへの深堀りは何？LLMへ適用されるハーネスの概念はそのままサブエージェントにも適用されるだけでは？フラクタル構造になっているイメージ」

**構造的含意**:
- Harness（親LLM実行環境）とサブエージェント実行環境は**自己相似**
- サブエージェント自身もLLMであり、プロンプト（≒CLAUDE.md相当）・rule・skill・hookを持ち得る
- 「サブエージェント目的関数」を**独立概念**として設計するのは過剰設計
- **親harnessの目的関数定義が、そのままサブエージェント階層にも再帰適用される**

**帰結（モデル更新）**:
- サブエージェント目的関数 = コンポーネント目的関数の1インスタンス（サブエージェントというコンポーネントの目的関数）
- 中間形（5-α）の判断も不要。**三部構造が一貫して適用される**
- 独立概念として「サブエージェント目的関数」を立てるのではなく、「サブエージェント = コンポーネントの一種」として統合する
- 階層を跨いで同じメカニズム（目的関数の三部構造・制御層分類・独立レビュー・Steering Loop）が再帰適用される
