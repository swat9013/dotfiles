## 2. 哲学理論カタログ

### A. 存在論・形而上学（8理論）

#### A1. プラトンのイデア論 → 抽象クラス/インターフェース

**概念**: 個物は理想的で不変の原型「イデア」の不完全な模倣。個別の馬は「馬らしさ」というイデアの影。

**設計への対応**:
- クラス（型）= イデア、インスタンス = 個物
- インターフェースを実装の前に定義する設計順序の根拠
- 多態性 = 複数の具体実装が共通のイデア的本質を表現

**設計指針**: 抽象層（インターフェース）を最優先で設計し、実装の揺らぎから構造を守る

#### A2. アリストテレスの質料形相論 → 属性と振る舞いの統合

**概念**: 個物は「質料（素材）」と「形相（規定原理）」の不可分な結合。プラトンと異なり形相は質料と分離不可能。

**設計への対応**:
- 質料 = フィールド（状態）、形相 = メソッド（振る舞い）
- 両者を一体化したカプセル化の哲学的根拠

**設計指針**: 「何をするか」（形相）を先に決め、必要な状態（質料）を後から決定する。データホルダーではなく行動主体としてモデリング

#### A3. アリストテレスの四原因説 → 責務駆動設計

**概念**: 存在の理由を4層で説明 — 質料因（何から）、形相因（どんな形）、作用因（誰が作った）、目的因（何のため）。

**設計への対応**:
- 質料因 = 技術選定、形相因 = アーキテクチャ、作用因 = 開発プロセス、目的因 = ビジネス上の責務

**設計指針**: 「なぜこのクラスが必要か」を目的因から言語化する。目的因 → 形相因 → 質料因の順で設計を逆算

#### A4. ライプニッツのモナド論 → カプセル化・疎結合

**概念**: 宇宙の最小単位「モナド」は「窓のない城」。外部と直接やり取りせず、「予定調和」で相互調整される。

**設計への対応**:
- モナド = 完全にカプセル化されたオブジェクト
- 「窓のない」= private フィールド
- 「予定調和」= メッセージパッシング、イベント駆動

**設計指針**: 各オブジェクトは自分の責務範囲で完全に独立。直接結合でなくイベント/インターフェース経由で協調

#### A5. メレオロジー（部分と全体） → 集約/コンポジション

**概念**: 部分と全体の関係を形式的に分析。どの階層で「部分」を定義するかで設計粒度が変わる。

**設計への対応**:
- 集約 = 部品が親消滅後もライフサイクル継続
- コンポジション = 部品が親と共に消滅
- マイクロサービス境界 = メレオロジー的な境界線定義

**設計指針**: ライフサイクルの独立性で集約/コンポジションを選択。集約ルートごとにサービス境界を引く

#### A6. ヘラクレイトス vs パルメニデス → 可変 vs イミュータブル

**概念**: ヘラクレイトス「万物流転」vs パルメニデス「存在は不変」。変化の本質性と恒常性の対立。

**設計への対応**:
- ヘラクレイトス的 = 可変状態、イベント駆動、状態遷移
- パルメニデス的 = イミュータブル設計、値オブジェクト、関数型

**設計指針**: イミュータブルな値オブジェクト + Event Sourcingで両者を止揚。「不変なイベントの流れ」から「現在の状態」を再構築

#### A7. 本質主義 vs 反本質主義 → 名目的型 vs 構造的型

**概念**: 本質主義: オブジェクトには本質的属性がある。反本質主義: カテゴリーは社会的構成、文脈依存。

**設計への対応**:
- 本質主義 = Nominal Typing（Java、C++の厳格な継承階層）
- 反本質主義 = Structural Typing / Duck Typing（Go、TypeScript）

**設計指針**: 「何であるか」より「何ができるか」で型を定義。Goの暗黙的インターフェース実装が反本質主義的設計の好例

#### A8. 存在論的コミットメント（クワイン） → ドメインモデル設計

**概念**: 理論を採用する時、その理論が前提する実体の存在を認めることになる。変数の量化範囲が存在論を決定。

**設計への対応**:
- ドメインモデル = システムが「存在する」と認める実体の完全リスト
- バウンディッドコンテキスト = 存在論的境界

**設計指針**: 「このシステムで何が存在するか」を明示的に定義。異なるコンテキストでは同じ「User」でも異なる存在論を持つ

### B. 認識論・構造主義・論理学（8理論）

#### B1. カントの認識論（現象と物自体） → 公開インターフェース vs 内部実装

**概念**: 我々が認識できるのは「現象」（感覚を通じて構成されたもの）のみ。「物自体」には到達できない。認識にはカテゴリー（先天的枠組み）が必要。

**設計への対応**:
- 現象 = 公開API/インターフェース（外部から見える姿）
- 物自体 = 内部実装の詳細
- カテゴリー = 型システム、スキーマ（認識の枠組み）

**設計指針**: API利用者は「物自体」（実装詳細）にアクセスすべきでない。スキーマ/型が「認識の枠組み」として機能し、データの解釈方法を規定する

#### B2. ソシュールの構造主義（差異の体系） → 命名・型体系

**概念**: 記号の意味はそれ自体にあるのでなく、他の記号との「差異」の関係で成立。シニフィアン（記号表現）とシニフィエ（記号内容）の恣意的結合。

**設計への対応**:
- 型システム = 差異の体系（`int`の意味は`string`でないことで定義される）
- 命名規則 = シニフィアンの体系（一貫した差異が意味を生む）

**設計指針**: 命名は他との差異が明確になるよう設計する。`getUser` vs `fetchUser` vs `loadUser`の差異を体系内で一貫させる

#### B3. ヴィトゲンシュタインの言語ゲーム → DSL・DDD

**概念**: 言語の意味は使用の中で決まる。異なる「言語ゲーム」（文脈）では同じ語が異なる意味を持つ。「意味＝使用」。

**設計への対応**:
- ユビキタス言語（DDD）= ドメイン固有の言語ゲーム
- DSL = 特定のゲームのための言語
- コンテキスト依存のAPI設計

**設計指針**: APIの「文法」はドメインコンテキストに合わせて設計する。同じ`create`でもドメインによって意味が異なることを許容し、バウンディッドコンテキストで分離

#### B4. ポパーの反証可能性 → テスト駆動設計

**概念**: 科学的理論は反証可能でなければならない。「すべての白鳥は白い」は黒い白鳥1羽で反証される。検証不能な理論は科学ではない。

**設計への対応**:
- テスト = 反証の試み
- テスト不能な仕様 = 反証不能な理論（価値が低い）
- TDD = 反証可能な仕様を先に書く方法論

**設計指針**: 仕様は「どうすれば失敗するか」が明確な形で書く。テスト不能な設計は曖昧な設計の兆候

#### B5. クーンのパラダイム論 → 設計パラダイム転換

**概念**: 科学は漸進的でなく、パラダイム転換（革命）を経て進む。通常科学（パラダイム内の問題解決）→ 異常の蓄積 → 危機 → 革命。

**設計への対応**:
- パラダイム = OOP / FP / リアクティブ等の設計思想
- 技術的負債の蓄積 = 異常の蓄積
- アーキテクチャ刷新 = パラダイム転換

**設計指針**: パラダイム内で解決困難な問題が蓄積したら、パッチではなくパラダイム転換（アーキテクチャ刷新）を検討すべき

#### B6. カリー＝ハワード対応 → 型安全性の論理的基盤

**概念**: 命題 = 型、証明 = プログラム。論理体系と型システムは同型。正しいプログラムを書くことは定理を証明することに等しい。

**設計への対応**:
- 型定義 = 命題の宣言
- コンパイル成功 = 証明完了
- 型エラー = 論理的矛盾の検出

**設計指針**: 型システムを「コンパイル時証明器」として活用。不正な状態を型レベルで表現不能にする（Make Illegal States Unrepresentable）

#### B7. ゲーデルの不完全性定理 → 型システムの限界

**概念**: 十分に強力な形式体系は、自身の無矛盾性を証明できない。真であるが証明不能な命題が必ず存在する。

**設計への対応**:
- 型システムでは捕捉できないバグが必ず存在する
- 完全な形式仕様は原理的に不可能
- 検証の多層化が必要（型 + テスト + レビュー + 監視）

**設計指針**: 単一の検証手段に依存しない。型チェック → ユニットテスト → 統合テスト → 運用監視の多層防御を設計

#### B8. パースの記号論 → API命名・直感的設計

**概念**: 記号を3分類 — アイコン（類似性: 🗑️）、インデックス（因果関係: 煙→火）、シンボル（慣習: 言語）。

**設計への対応**:
- アイコン的API = 操作対象と名前が直観的に類似（`file.delete()`）
- インデックス的API = 因果関係を示唆（`logger.warn()` → 問題の予兆）
- シンボル的API = 慣習に依存（`i`がループ変数）

**設計指針**: アイコン的命名（直感で分かる）を優先。シンボル的命名は学習コストが高いため、広く定着した慣習のみ採用

### C. 倫理学・美学・プラグマティズム（10理論）

#### C1. カントの定言命法 → API一貫性テスト

**概念**: 「汝の格率が普遍的法則となるよう行為せよ」。行為の道徳性は結果でなく、普遍化可能性で判断される。

**設計への対応**:
- API設計パターンの普遍化テスト: 「すべてのAPIがこのパターンに従えるか？」
- 例外的扱いは設計の不整合の兆候

**設計指針**: 新しいAPIパターンを導入する際、「これをシステム全体に適用しても破綻しないか？」を自問する

#### C2. ロールズの正義論（無知のヴェール） → 利用者視点のAPI設計

**概念**: 自分の社会的立場を知らない状態（無知のヴェール）で公正な制度を設計する思考実験。

**設計への対応**:
- 自分がAPIの提供者か消費者か分からない状態で設計する
- 初心者/上級者、Web/Mobile/Backend等、あらゆる利用者を想定

**設計指針**: 「自分が最も不利な立場の利用者だったら、このAPIを使えるか？」をテストする

#### C3. 功利主義（ベンサム/ミル） → トレードオフ最適化

**概念**: 最大多数の最大幸福。行為の正しさは結果（全体の幸福量）で判断される。

**設計への対応**:
- パフォーマンス vs 可読性のトレードオフ判断
- 多くの利用者にとっての「幸福」（利便性）を最大化

**設計指針**: トレードオフは影響を受ける利用者数×影響度で定量的に判断する。少数の上級者向け最適化より多数の一般利用者のDXを優先

#### C4. デューイのプラグマティズム → YAGNI・実用優先

**概念**: 思考は問題解決のツール。理論の真偽は実践的帰結で判断される。「やりながら学ぶ」。

**設計への対応**:
- YAGNI原則の哲学的根拠
- 推測的設計より、実際の問題への対応を優先

**設計指針**: 将来のための設計ではなく、今の問題を解く設計を選ぶ。抽象化は3つ目の具体例が出てから

#### C5. オッカムの剃刀 → 最小限設計

**概念**: 説明に必要以上の仮定を導入すべきでない（思考節約の原理）。

**設計への対応**:
- 不要な概念・中間層・抽象化の排除
- 同じ問題を解くなら、よりシンプルな設計を選ぶ

**設計指針**: 新しいクラス/レイヤーを追加する前に「これなしで解決できないか？」を問う。3行のコード重複は早すぎる抽象化より優れる

#### C6. 侘び寂び・間（ま） → 余白のある設計

**概念**: 不完全さ、不規則さ、空間の価値を重視する東洋美学。完成されすぎたものより、余白があるものに美を見出す。

**設計への対応**:
- API拡張ポイント = 意図的な「余白」
- 完全に定義されたAPIより、拡張可能な余白を持つAPIが長命
- 不完全さの受容 = MVP思考

**設計指針**: 最初から完璧を目指さず、拡張可能な余白（extensionフィールド、プラグイン機構）を残す

#### C7. ヘーゲルの弁証法 → リファクタリングサイクル

**概念**: テーゼ（正）→ アンチテーゼ（反）→ ジンテーゼ（合）の三段階で概念が発展。対立は廃棄でなく止揚（Aufheben）される。

**設計への対応**:
- 初期設計（テーゼ）→ 問題発覚（アンチテーゼ）→ リファクタリング（ジンテーゼ）
- 設計上の矛盾は進化の契機

**設計指針**: 設計の矛盾を「修正すべきバグ」でなく「進化の機会」として捉える。対立する要件は止揚（より高次の統合）を目指す

#### C8. 老子の道・無為自然 → Convention over Configuration

**概念**: 人為的努力を避け、自然の流れに従う。最良の統治は民が統治されていると感じないもの。

**設計への対応**:
- Convention over Configuration（設定より規約）
- 良いフレームワークは存在を感じさせない
- デフォルト値の充実

**設計指針**: 80%のユースケースを設定不要でカバーするデフォルトを設計する。カスタマイズは例外的なケースのみ

#### C9. アリストテレスの中庸 → 抽象化レベルの最適化

**概念**: 徳は過度と不足の中間にある。中庸は機械的な中間点でなく、状況ごとの実践的知恵（フロネシス）で判定。

**設計への対応**:
- 過度な抽象化（AbstractFactoryFactoryFactory）と具象べた書きのバランス
- 適切なレイヤー数、クラス粒度の判断

**設計指針**: 抽象化は「過度」と「不足」の両極を意識し、文脈に応じた最適点を実践知で判断する。一般的目安: レイヤー3-5層、クラス責務3-5個

#### C10. 社会契約論 → Design by Contract

**概念**: 社会の成立根拠を構成員間の相互契約に求める。権利と義務は契約で明示的に定義される。

**設計への対応**:
- 前提条件（precondition）= 呼び出し側の義務
- 事後条件（postcondition）= 提供側の義務
- 不変条件（invariant）= 契約の基盤

**設計指針**: API境界では前提条件・事後条件・不変条件を明示的に定義する。契約違反時の責任所在（呼び出し側 vs 提供側）を明確にする

### D. 意味論（6理論）

#### D1. フレーゲの意義と意味（Sinn/Bedeutung） → 命名設計

**概念**: 同じ対象（Bedeutung = 意味/指示対象）を指す表現でも、「記述の仕方」（Sinn = 意義）が異なりうる。「明けの明星」と「宵の明星」は同じ金星を指すが、意義が異なる。

**設計への対応**:
- `userId` と `authorId` は同じIDを指しうるが、記述の仕方（文脈での役割）が異なる
- 命名は「何を指すか」だけでなく「どう指すか」（どの文脈・役割から参照するか）も設計判断

**設計指針**: 同一対象への複数の参照が存在する場合、各名前が「どの文脈での役割」を表現しているか明確にする。エイリアスの安易な導入は意義の曖昧化を招く

#### D2. 合成性原理（フレーゲ） → 関数合成・モジュール設計

**概念**: 全体の意味は部品の意味とその結合方法の関数として決定される。部品の意味を知っていれば、全体の意味を予測できる。

**設計への対応**:
- メソッドチェーン・パイプラインは合成的（各ステップの意味から全体を予測可能）
- 副作用やグローバル状態への依存は合成性を破壊する（部品の意味だけでは全体を予測不能）

**設計指針**: 部品を入れ替えても全体の意味が予測可能な設計を目指す。合成性を破る要素（暗黙の状態変更、順序依存）を最小化する

#### D3. 発話行為論（オースティン/サール） → CQS原則

**概念**: 発話は「事実を述べる」（事実確認的 = constative）だけでなく「行為を遂行する」（遂行的 = performative）こともある。「この会議は閉会します」は発話自体が行為。

**設計への対応**:
- Query = 事実確認的（`getUser()` は状態を述べる）
- Command = 遂行的（`deleteUser()` は行為を実行する）
- CQS（Command-Query Separation）の哲学的根拠

**設計指針**: 一つのメソッドが「事実を述べる」と「行為を遂行する」を混在させない。混在は呼び出し側に予期しない副作用をもたらす

#### D4. 可能世界意味論（クリプケ） → 状態空間・Property-based Testing

**概念**: 命題の真偽は現実世界だけでなく「可能なすべての世界」で評価される。必然的真理 = すべての可能世界で真。

**設計への対応**:
- 型の安全性 = すべての可能な入力状態で不変条件が成立するか
- Property-based testing = 可能な入力空間全体に対する性質の検証
- Model checking = 可能な状態遷移すべてで性質が保持されるか

**設計指針**: 「たまたま動く」ではなく「あらゆる可能な状態で成立する」性質を定義し検証する

#### D5. 内包と外延（カルナップ） → インターフェースの意図区別

**概念**: 外延 = ある概念が実際に適用される対象の集合。内包 = その概念の定義・意味内容。外延が同じでも内包は異なりうる（「羽のない二足歩行動物」と「人間」）。

**設計への対応**:
- 外延 = あるインターフェースを実装するクラスの集合
- 内包 = そのインターフェースの定義・意図
- `Serializable` と `Loggable` が偶然同じクラス群に適用されても、内包（設計意図）は異なる

**設計指針**: インターフェースは外延（たまたま同じ実装群）でなく内包（設計意図）で分離する。偶然の一致でインターフェースを統合しない

#### D6. プロトタイプ理論（ロッシュ） → 継承階層の限界

**概念**: カテゴリーは古典的な必要十分条件ではなく、典型例（プロトタイプ）からの類似度で構成される。「鳥」のプロトタイプはスズメであり、ペンギンは周辺的メンバー。

**設計への対応**:
- 継承階層は「すべてのメンバーが共通属性を持つ」という古典的カテゴリー観に基づく
- 現実のドメインはプロトタイプ的（周辺的メンバーが共通属性を満たさない）
- ペンギン問題: `Bird.fly()` をすべてのBirdサブクラスに強制できない

**設計指針**: 深い継承階層より合成（Composition）を優先する根拠。「典型例」で設計したインターフェースが周辺的メンバーで破綻しないか検証する

## 4. 推奨事項

### 実務への適用アプローチ

これらの哲学理論は「正解を導く公式」ではなく「問いを立てるレンズ」として使う。

1. **設計レビュー時のチェックリスト**: 統合フレームワーク表の「問い」を設計レビューの観点に組み込む
2. **対立の止揚**: 「可変vs不変」「抽象vs具象」「完全性vs拡張性」といった対立を二者択一でなく、弁証法的に統合する視点を持つ
3. **段階的適用**: まずは四原因説（目的因からの設計）とオッカムの剃刀（最小限設計）の2つから始めるのが最も投資対効果が高い

### 注意点

- 哲学的類推は思考の補助であり、設計の正当化に使うべきでない（「プラトンがそう言っているから」は根拠にならない）
- 同じ問題に対して矛盾する哲学的助言が得られることがある（本質主義vs反本質主義等）。これは「どちらが正しいか」ではなく「この文脈ではどちらが適切か」の問い
