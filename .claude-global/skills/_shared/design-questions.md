# 設計判断の問い

設計フェーズごとに立てるべき問い。「正解を導く公式」ではなく「思考のレンズ」として使用する。
詳細（各理論の解説・出典）: `docs/philosophy-for-software-design.md`

## 適用ルール

1. **全項目を走査しない**。対象コードに関連する問いのみ選択する
2. **「問題なし」は書かない**。該当しない問いはスキップする
3. 検出した問題は「深掘り」に従い、**条件・因果・代替案**まで展開する

## ドメインモデリング

- **存在論的コミットメント**（理論が前提する実体の存在を認めることになる — クワイン）
  問い: このシステムで「何が存在する」と認めるか？不要な概念を持ち込んでいないか？
  検出: テーブル/クラスに対応するビジネス概念を利害関係者が説明できない
  深掘り: その概念を削除したとき、どのユースケースが壊れるか？壊れないなら不要

- **言語ゲーム**（言語の意味は使用の中で決まる。異なる文脈では同じ語が異なる意味を持つ — ヴィトゲンシュタイン）
  問い: この語はこの文脈で何を意味するか？別コンテキストと混同していないか？
  検出: 同じ用語が複数のモジュール/チームで異なる意味で使われている
  深掘り: 意味のズレが実行時エラーやデータ不整合として表面化する経路はあるか？

- **バウンディッドコンテキスト**（言語ゲームの境界をシステム設計に適用 — エヴァンス/DDD）
  問い: 同じ名前の概念がコンテキスト間で異なる意味を持つなら、分離すべきか？
  検出: 1つのモデルが複数のユースケースに引きずられてフィールドが肥大化している
  深掘り: 分離した場合の変換コスト vs 混同が引き起こすバグの発生確率と修正コスト

## クラス・モジュール設計

- **四原因説 + オッカムの剃刀**（存在の理由を目的因から説明せよ。目的因が言語化できないものは不要な仮定 — アリストテレス/オッカム）
  問い: このクラスの存在理由をビジネス責務で言語化できるか？できないなら除去できないか？
  検出: 「将来必要になるかも」が唯一の導入根拠 / 責務を1文で説明できない
  深掘り: このクラスを削除して呼び出し元に処理をインライン化した場合、可読性・テスト容易性は悪化するか？

- **質料形相論**（個物は質料=素材と形相=規定原理の不可分な結合 — アリストテレス）
  問い: 属性（状態）と振る舞い（メソッド）は一体として意味を成すか？データホルダーになっていないか？
  検出: getterのみでロジックがない / フィールドに実体と異なる値を設定している
  深掘り: その乖離はいつ・どこで表面化するか？呼び出し側が乖離を意識する必要が生じる条件は？

- **メレオロジー**（部分と全体の関係を形式的に分析。ライフサイクルの独立性で関係性が決まる）
  問い: 部品のライフサイクルは親と同一か独立か？（→ コンポジション or 集約の選択）
  検出: 親を削除しても子が残るべきケースでコンポジション、または親と運命を共にすべきケースで集約を使っている
  深掘り: ライフサイクルの不一致がデータ整合性やリソースリークを引き起こすシナリオは？

- **ヘラクレイトス vs パルメニデス**（万物流転 vs 存在は不変。変化の本質性と恒常性の対立）
  問い: この状態は可変であるべきか、不変であるべきか？変化の記録が必要か？
  検出: 可変状態を複数箇所から同時更新している / 変更履歴が必要なのにスナップショットのみ
  深掘り: イミュータブルにした場合のコピーコスト vs 可変にした場合の並行アクセス制御コスト

## 型・インターフェース設計

- **本質主義 vs 反本質主義**（本質的属性でカテゴリーを定義する vs カテゴリーは文脈依存の構成物）
  問い: 「何であるか」で型付けるか？「何ができるか」で型付けるか？
  検出: 継承階層が「周辺的メンバー」（ペンギン問題）で破綻している / 構造が同じだけで型を統合している
  深掘り: 名目的型付けの厳格さが必要な安全境界はどこか？構造的型付けの柔軟性が必要な拡張ポイントはどこか？

- **カリー＝ハワード対応**（命題=型、証明=プログラム。型システムは証明器として機能する）
  問い: 不正な状態を型レベルで表現不能にできるか？
  検出: 実行時バリデーションで弾いている不正状態が、型定義では許容されている
  深掘り: 型で表現不能にするコスト（型の複雑化）vs 実行時検証で済ませるリスク（検証漏れ）

- **現象 vs 物自体**（認識できるのは現象のみ。物自体には到達できない — カント）
  問い: 公開インターフェースは内部実装から十分に分離されているか？
  検出: 内部のデータ構造・状態遷移が公開APIの型シグネチャに露出している
  深掘り: 内部実装を変更したとき、公開インターフェースの利用者に破壊的変更が波及するか？

- **差異の体系**（記号の意味は他の記号との差異の関係で成立する — ソシュール）
  問い: 命名は体系内で差異が明確か？類似名が混乱を生んでいないか？
  検出: `getX` / `fetchX` / `loadX` の使い分け基準が不在 / 同一スコープ内で類似名が衝突している
  深掘り: 命名体系にルールを追加したとき、既存の名前との整合性は保たれるか？

- **Sinn/Bedeutung + 内包/外延**（同じ対象を指す表現でも記述の仕方が異なりうる。外延が同じでも内包=設計意図が異なれば別物 — フレーゲ/カルナップ）
  問い: 同じ対象を指す名前、または同じ実装群を持つインターフェースを、意図の違いを無視して統合していないか？
  検出: `userId`と`authorId`が型エイリアスなしで同一型 / 偶然同じクラス群が実装する2つのインターフェースを統合している
  深掘り: 意図が分岐する将来の変更シナリオはあるか？統合を解く際のコストはどの程度か？

## API設計

- **定言命法**（汝の格率が普遍的法則となるよう行為せよ — カント）
  問い: このパターンをシステム全体に適用しても破綻しないか？
  検出: 例外的な扱い・特別ルートが存在する / 「ここだけ特殊」という説明が必要
  深掘り: 特殊扱いが増殖した場合、システムのどの性質（テスト容易性、一貫性、学習コスト）が最初に破綻するか？

- **無知のヴェール**（自分の立場を知らない状態で公正な制度を設計する思考実験 — ロールズ）
  問い: 自分が最も不利な立場の利用者だったら、このAPIを使えるか？
  検出: ドキュメントを読まないと正しく使えない / エラーメッセージが原因特定に不十分
  深掘り: 最も経験の浅い利用者が最初に躓くポイントはどこか？その躓きは設計で防げるか？

- **社会契約論**（権利と義務は契約で明示的に定義される → Design by Contract）
  問い: 前提条件・事後条件・不変条件は明示的に定義されているか？
  検出: 呼び出し側と提供側で「誰が何を保証するか」の認識がずれている
  深掘り: 契約違反が発生したとき、原因箇所の特定にどれだけの時間がかかるか？

- **記号論**（記号を3分類: アイコン=類似性、インデックス=因果、シンボル=慣習 — パース）
  問い: 命名は直感的か？慣習に過度に依存していないか？
  検出: メソッド名単体から振る舞いを推測できない / プロジェクト固有の略語が定義なしで使われている
  深掘り: アイコン的命名（直感で分かる）に置き換え可能か？シンボル的命名なら慣習が十分に定着しているか？

- **発話行為論**（発話は事実を述べるか行為を遂行するか。混在させるべきでない — オースティン/サール → CQS）
  問い: このAPIは事実を返すか、行為を遂行するか？混在していないか？
  検出: 副作用のあるgetterがある / 戻り値のあるコマンドがある
  深掘り: 混在を分離した場合、呼び出し側のコードはどう変わるか？分離コストは正当化できるか？

- **合成性原理**（全体の意味は部品の意味とその結合方法から決定される — フレーゲ）
  問い: 部品を組み合わせた全体の意味は、部品の意味から予測可能か？
  検出: メソッドチェーンの途中で暗黙の状態変更がある / 実行順序を変えると結果が変わるが型では制約できていない
  深掘り: 合成性を破っている要素（暗黙の依存、グローバル状態）を除去するコスト vs 放置した場合のデバッグコスト

## 設計判断・トレードオフ

- **中庸**（徳は過度と不足の中間。状況ごとの実践的知恵で判定する — アリストテレス）
  問い: 過度な抽象化と具象べた書きのどちらに偏っていないか？
  検出: AbstractFactoryFactory的な多層抽象 / 逆に3箇所以上の明白な重複を放置
  深掘り: 現在の抽象レベルを1段上げた場合/下げた場合、コードの変更容易性はどう変わるか？

- **プラグマティズム**（理論の真偽は実践的帰結で判断される。今の問題を解くことが優先 — デューイ）
  問い: 将来のための設計ではなく、今の問題を解いているか？
  検出: 現時点で呼び出し元が1つしかない汎用インターフェース / 実際のユースケースがないextensionポイント
  深掘り: 「将来の拡張性」を削除した場合、追加が必要になったときの実装コストは本当に高いか？

- **無為自然**（人為的努力を避け自然の流れに従う。最良の統治は存在を感じさせない — 老子）
  問い: 80%のユースケースを設定不要でカバーするデフォルトがあるか？
  検出: 全利用者に設定を強制している / デフォルト値がなく、ゼロ値が不正な状態を生む
  深掘り: デフォルトが「間違っている」場合の影響範囲 vs 設定を強制する場合の導入コスト

- **弁証法**（テーゼ→アンチテーゼ→ジンテーゼ。対立は止揚される — ヘーゲル）
  問い: 設計上の矛盾は「修正すべきバグ」か「進化の機会」か？
  検出: 対立する要件をどちらか一方の犠牲で解決しようとしている / 同じ箇所で繰り返し設計変更が発生
  深掘り: 両方の要件を満たす「より高次の統合」は存在するか？それは複雑さに見合うか？

## 検証戦略

- **反証可能性**（科学的理論は反証可能でなければならない — ポパー）
  問い: 仕様は「どうすれば失敗するか」が明確な形で書かれているか？
  検出: テストが正常系のみ / 境界条件・異常系のテストケースが欠落
  深掘り: この仕様を反証するための最小限のテストケースは何か？

- **不完全性定理**（十分に強力な形式体系は自身の無矛盾性を証明できない — ゲーデル）
  問い: 単一の検証手段に依存していないか？（型 + テスト + レビュー + 監視の多層防御）
  検出: 「型があるからテスト不要」「テストがあるから監視不要」という判断
  深掘り: 現在の検証手段が見逃すバグのカテゴリは何か？それを補う層はあるか？

- **パラダイム論**（科学は漸進的でなくパラダイム転換を経て進む。異常の蓄積が危機を招く — クーン）
  問い: パラダイム内で解決困難な問題が蓄積していないか？パッチではなく転換が必要か？
  検出: 同じアーキテクチャ上のワークアラウンドが3つ以上蓄積 / 新機能追加のたびに既存部分が壊れる
  深掘り: ワークアラウンドの除去に必要な構造変更の範囲は？段階的な移行パスはあるか？
