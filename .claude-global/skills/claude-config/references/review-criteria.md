## モード別の適用

| 観点 | 診断モード | 自己改善モード |
|------|-----------|--------------|
| 一貫性 | 適用した変更がプロジェクト既存設定と矛盾しないか | references/rules 間の重複・矛盾 |
| 正確性 | 設定キー・値・パスが正しいか | 記述が現行の Claude Code 動作と一致するか |
| 形式準拠 | 編集先ファイルの既存フォーマットに合っているか | 行数制限・TOC・見出し階層 |
| 実用性 | 改善が実際に効果をもたらすか | Claude の既知情報を冗長に書いていないか |

## 一貫性

### チェック項目

- **既存設定との整合**: 適用した変更がプロジェクトの他の設定と矛盾しないか
- **references/rules 重複なし**: 同じ情報が両方に存在しないか（自己改善モード）
- **矛盾なし**: references と rules で相反する記述がないか（自己改善モード）
- **用語統一**: 同一概念に異なる用語を使っていないか
- **CLAUDE.md との整合**: 上位の CLAUDE.md と矛盾しないか

### フラグ基準

- 適用した変更が既存の hooks/permissions と競合
- references/ と rules/ に同一情報の重複
- 同一設定項目に対する矛盾した推奨値
- CLAUDE.md の方針に反する記述の追加

## 正確性

### チェック項目

- **設定キー・値の正確性**: JSON キー名、許容値が正しいか
- **パス参照の正確性**: ファイルパスが実在するか
- **現行動作との整合**: 記述が Claude Code の実際の動作と一致するか
- **バージョン依存情報**: 廃止・変更された機能を参照していないか

### フラグ基準

- 存在しない設定キーや値の記述
- 実在しないファイルパスへの参照
- 廃止された機能の推奨

## 形式準拠

### チェック項目

- **既存フォーマット維持**: 編集先ファイルの表/リスト/見出し構造を壊していないか
- **行数制限**: references/ のファイルが 200 行以下か（自己改善モード）
- **TOC**: 100 行超のファイルに TOC があるか（自己改善モード）
- **見出し階層**: 一貫した Markdown 構造か

### フラグ基準

- 編集により既存の表構造やリスト構造が崩れた
- 200 行超過のファイル
- 100 行超で TOC 欠落
- 見出しレベルの飛び（h2 → h4 等）

## 実用性

### チェック項目

- **改善効果**: 適用した変更が実際に動作改善につながるか（診断モード）
- **既知情報の排除**: Claude の一般知識に含まれる情報を冗長に記述していないか（自己改善モード）
- **具体性**: 抽象的な助言でなく、具体的な判断基準・値・例があるか
- **日付のハードコード**: 腐りやすい具体日付を含んでいないか

### フラグ基準

- 適用した設定が実際には効果のない値（診断モード）
- Claude が標準知識として持つ情報の列挙（自己改善モード）
- 「適切に設定する」のような曖昧な記述
- 相対表現で代替可能なハードコード日付

## 高信号フィルタ

### フラグすべき

| カテゴリ | 診断モード例 | 自己改善モード例 |
|---------|------------|---------------|
| 不正確 | 存在しない設定キーの適用 | 廃止機能の推奨記述 |
| 矛盾 | 既存 hooks との競合 | references/rules の相反記述 |
| 破壊 | 既存設定の意図しない上書き | 既存セクション構造の破壊 |
| 重複 | - | references/rules 間の同一記述 |
| 冗長 | - | Claude 既知情報の繰り返し |

### フラグしない

| カテゴリ | 理由 |
|---------|------|
| 表現の好み | 主観的（内容が正確なら十分） |
| 既存ファイルの問題 | 今回の変更スコープ外 |
| 構成の好み | 動作に影響しない配置変更 |
