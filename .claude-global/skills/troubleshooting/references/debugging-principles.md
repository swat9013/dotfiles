# デバッグ原則リファレンス

## 構造化された仮説フォーマット

仮説立案時は以下のフォーマットで記録:

```
仮説: [原因の推定]
予測: この仮説が正しければ [検証可能な予測]
反証: この仮説が正しければ起こりえない [反証条件]
検証方法: [具体的な調査手順]
結果: [確認 / 棄却] (証拠: [...])
```

## 分割統治の適用パターン

| 対象空間 | 手法 | ツール |
|---------|------|--------|
| コード空間 | コメントアウト/条件分岐で半減 | Read + Edit |
| コミット履歴 | 二分探索で問題導入コミットを特定 | `git bisect` |
| 入力データ | 要素を半減して再現テスト | テストケース |
| コンポーネント | モック/スタブで依存を切り離し | テストフレームワーク |

## バグの3層モデル

問題が解消しない場合、層を切り替えて考える:

1. **コードの問題**: 構文・ロジックエラー → 即時修正
2. **環境の問題**: 依存関係・設定・リソース → 環境修正
3. **理解の問題**: 仕様の誤解、言語/FWの動作理解不足 → 仕様・前提の見直し

第3層が最も見落とされやすい。「コードは正しいが前提が間違っている」パターン。

## AI向け認知バイアス対策

| バイアス | 兆候 | 対策 |
|---------|------|------|
| 確証バイアス | 仮説を支持する証拠ばかり探している | 反証テストを先に実行する |
| 機能的固着 | 同じツール/手法を3回以上繰り返している | 手法を切り替える（別ツール、別観点） |
