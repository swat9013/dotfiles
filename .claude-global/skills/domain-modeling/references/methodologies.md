## DDD戦略的設計

Bounded Context・Context Map・Ubiquitous Language でドメインを構造化する手法。ビジネス戦略とソフトウェア設計の整合性を実現する。

### 主要概念

| 概念 | 役割 |
|------|------|
| Ubiquitous Language | ドメインエキスパートと開発者が共有する明確な語彙 |
| Bounded Context | ドメインモデルが有効で一貫性を保つ意味論的な境界 |
| Aggregate | トランザクション一貫性を保つ関連オブジェクトのグループ |
| Domain Event | ドメインで発生した意味のある出来事を表す不変のファクト |
| Context Map | Bounded Context間の関係性を図式化したもの |

### 利点

- マイクロサービス分割の自然な境界を提供
- チーム間の認識ズレを減らす
- ドメイン知識の散逸を防止

### 欠点

- 導入時のドメイン分析にコストが必要
- 小規模プロジェクトでは過剰になりうる
- Context Map の維持に継続的な努力が必要

## Event Storming

Alberto Brandolini（2012年）創案のグループモデリング手法。ワークショップ形式でドメインイベントを時系列に可視化する。

### 3段階

| レベル | 目的 | 参加者 | 所要時間 |
|--------|------|--------|----------|
| Big Picture | ドメイン全体のイベント流を把握、境界発見 | 5-20人 | 数時間〜1日 |
| Process Level | 特定プロセスのコマンド・イベント・集約を詳細化 | 3-10人 | 数時間 |
| Design Level | タクティカルDDDパターンの実装設計 | 3-5人（開発者中心） | 数時間 |

### 利点

- 数時間で包括的なドメインモデルを構築可能
- ドメインエキスパート・開発者の共通理解を即時構築
- イベント・コマンド・集約を自然に抽出

### 欠点

- 参加者全員の時間確保が必要
- ファシリテーターのスキルに成否が大きく依存
- 出力の記録・整理に別途努力が必要

## Domain Storytelling

Stefan Hofer & Henning Schwentner による手法。ドメインエキスパートの物語を象形図言語で記録する。

### 進め方

1. ドメインエキスパートにビジネスプロセスを「物語」として説明させる
2. アクター・モノ・活動・相互作用を象形図言語で記録
3. 記録過程で曖昧性・矛盾を質問により明確化
4. 繰り返し検証し、Ubiquitous Language を抽出

### 利点

- 物語形式で自然にドメイン知識を引き出せる
- 低学習コスト、Ubiquitous Language 構築に直結
- 技術チーム・ビジネスチーム間の橋渡しに優れる

### 欠点

- Event Storming より速度は劣る（複数セッション必要）
- 大規模ドメインでは図が複雑化する可能性
- イベント因果関係や並行処理の表現が限定的

### ツール

- [WPS Community Tool](https://domainstorytelling.org/)

## Event Modeling

Adam Dymitruk 開発の手法。時間軸上でユーザー操作・イベント・ビュー状態を統一表記で記述する。

### 構成要素

| 要素 | 役割 |
|------|------|
| タイムライン | 時間軸（横軸）にイベントシーケンスを配置 |
| ユーザー入力 | UIからのコマンド発行 |
| イベント | 状態変化を表す不変のファクト |
| ビュー/Read Model | イベントから派生する読み取り用データ |
| プロセッサ | 自動化処理（スケジュール、外部連携） |

### 利点

- 時間軸でシステム動的挙動が直感的に把握可能
- Event Sourcing/CQRS との親和性が高い
- 自動化処理を明示的にモデル化可能

### 欠点

- 学習曲線が急
- 大規模では図の分割管理が必要
- イベント駆動アーキテクチャの前提知識を要求

### リソース

- [eventmodeling.org](https://eventmodeling.org/)

## Example Mapping

Cucumber チームが提唱する手法。4色カードでビジネスルール・具体例・エッジケースを捉える。

### 4色カード

| 色 | 内容 |
|----|------|
| 黄 | ルール（ビジネスルール） |
| 青 | 具体例（ルールを示す例） |
| 赤 | 想定（不確実な前提） |
| ピンク | 質問（未解決の疑問） |

### 進め方

- 小グループ（Three Amigos: 開発者・テスタ・ビジネスアナリスト）で会話
- 25分で理解できなければストーリーが大きすぎると判定 → 分割
- BDDフレームワーク（Cucumber等）で自動テスト化

### 利点

- 小グループ（3人）で迅速（25分）に進行
- エッジケース・受入基準の発見に優れる
- BDDフレームワークと直結

### 欠点

- 戦略的設計の判断には不向き
- 個別ユースケース単位に限定
- ドメインモデルへの統合は別途必要

## 手法間の関係性

```
戦略的設計 ──────────────────────── タクティカル設計 ────── 検証

Event Storming (Big Picture)     Event Storming (Process/Design)
        │                                │
        ├─── 全体のイベント流を把握       ├─── コマンド・集約を具体化
        │                                │
Domain Storytelling              Event Modeling
        │                                │
        ├─── 詳細知識を物語から抽出       ├─── 時系列で状態遷移を詳細化
        │                                │
        ▼                                ▼
DDD 戦略的設計                   Example Mapping (BDD)
        │                                │
        └─── Bounded Context 確定        └─── 受入基準を自動テスト化
```

### ドメイン複雑度別の推奨

| 複雑度 | 推奨手法 |
|--------|---------|
| 低（CRUD中心） | Example Mapping のみで十分 |
| 中（ビジネスルール有） | Event Storming (Big Picture) + DDD + Example Mapping |
| 高（複数チーム・大規模） | 全手法を段階的に組み合わせ |
