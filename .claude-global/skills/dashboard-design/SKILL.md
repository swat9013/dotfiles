# Dashboard Design

## 1. Decision Test — メトリクス搭載可否の3問

各メトリクスを搭載する前（設計）または既存メトリクスの存在理由を問う（診断）際に適用する。

| 問 | 問いかけ | 答えられない場合の判定 |
|---|---------|---------------------|
| Q1 | この数値が動いたら**誰が何をする**？ | Vanity metric → 削除 |
| Q2 | どの**OKR/KPI に寄与**する？ | 支援指標に降格（Overview から外す） |
| Q3 | この数値の**所有者**は誰？ | 実装・運用不可 → 実装見送り |

**設計時**: 3問すべてに答えられるものだけ搭載する。
**診断時**: 全問に答えられないメトリクスを「要対応」としてマークする。

詳細な判定ロジック → `references/design-guide.md`

## 2. 3A 評価マトリクス（Eric Ries）

| 基準 | 検証問い | OK の条件 |
|------|---------|----------|
| **Actionable** | 「X を変えると Y が動く」と言えるか | 特定アクションと結果の因果関係が明確 |
| **Accessible** | 非技術者が 3 秒で解釈できるか | 統計知識なしで即座に判断可能 |
| **Auditable** | データソースを追跡・再現できるか | 数値の出所が明確で検証可能 |

**設計時（自己評価）**: 搭載前に 3 基準すべてを満たすか確認する。
**診断時（採点）**: 各メトリクスに `OK / 改善余地あり / 要対応` + 理由 1 行でラベルを付ける。

## 3. アンチパターン早見表

| アンチパターン | 症状 | 根本原因 | 修正方向 |
|-------------|------|---------|---------|
| **Metric Overload** | 誰も全部見ない | 追加基準なしで積み増し | Decision Test で篩にかけ 5–9 個に限定 |
| **文脈なし数値** | 上下どちらが良いか判断不可 | 比較軸の欠如 | 目標値・前期間比・トレンドを必須化 |
| **Vanity 優先配置** | 総数が大きく表示されている | 見栄え優先の配置 | 3A テストで搭載可否を判定し降格 |
| **診断能力欠落** | 異常は見えるが「なぜ」を追えない | ドリルダウン未設計 | Layer 3 に根本原因分析パスを設ける |
| **更新停止** | データが古い・所有者不在 | ライフサイクル未定義 | 四半期レビュー（ASSESS→PLAN→ACT→REVIEW）を設計に組み込む |

**アラート設計の核心**: 静的閾値は成長・季節変動で陳腐化する。7 日移動平均 ±2σ などの動的閾値を優先し、静的閾値は補助に留める。

詳細な診断採点基準 → `references/diagnosis-rubric.md`

## 4. 診断フロー概要

既存ダッシュボードを診断する際は以下の 4 段階で進める。診断の目的は「評価・ラベル付け」ではなく「次のアクションを明確にすること」。

**Gotcha**: スコアリングや合否判定が目的化すると、改善議論より防衛的なコミュニケーションを生む。各段階の出力は常に「次に何をするか」に接続する。

### Stage 1: 現状把握
- 表示されているメトリクスを列挙する
- Leading / Lagging、運用層（日次）/ 実行層（週次）/ 戦略層（月次〜）でレイヤー分類する

### Stage 2: 目的確認
- Decision Test の 3 問を各メトリクスに適用する
- 3 問すべてに答えられないメトリクスを「存在理由不明」としてマークする

### Stage 3: 3A 評価
- 各メトリクスに `OK / 改善余地あり / 要対応` ラベル + 理由 1 行を付ける
- Actionable・Accessible・Auditable のうち欠けている基準を特定する

### Stage 4: 改善アクション
- アンチパターン早見表と照合し、該当するパターンを特定する
- 最大 3 件の改善アクションを提示し、優先順位の根拠を明示する（影響範囲・対応コスト）
