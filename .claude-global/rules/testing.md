# テスト設計原則

テストコードが表現すべきは **What**（何を期待するか）。

## Khorikov 4本柱（優先順）

1. **リファクタリング耐性** — 実装変更で壊れない（偽陽性排除）
2. **保護性** — バグを検出できる（偽陰性排除）
3. **保守性** — 読みやすく変更しやすい
4. **速度** — 高速に実行できる

## 設計ルール

- 観察可能な振る舞いを検証する（実装詳細ではなく）
- Mockはプロセス外依存（外部API, メール送信等）のみ。Stubのverifyは禁止
- DBはモックしない（実装詳細）。統合テストで実DBを使用
- 1テスト1アクション。AAA構造（Arrange / Act / Assert）を徹底
- カバレッジ率を目的化しない。形骸的なモック検証でカバレッジを稼がない
- ホスト環境を書き換えない。破壊操作は tempdir / テスト専用DB / サンドボックスに隔離し、afterEach で失敗時もクリーンアップする

## テストケース設計

- 正常系・異常系・境界値を網羅
- テスト名で仕様を表現: `[対象]_[条件]_[期待結果]`
- テストは独立して実行可能に
- 以下5観点で入出力・振る舞いパターンを抽出する

| 観点 | 抽出内容 |
|------|---------|
| 境界値 | 最小/最大、off-by-one、空、null/undefined、0/負数 |
| 異常値 | 型不正、範囲外、不正フォーマット、巨大入力 |
| 状態遷移 | 初期→各終端、不正遷移の拒否、同一操作の冪等性 |
| 偽陰性対策（保護性） | バグが通過しうる経路を列挙し各々に検知ケース |
| 偽陽性対策（リファクタリング耐性） | 実装詳細依存の脆い検証を排除 |

## 冪等性・独立性

| 依存源 | 遮断方法 |
|--------|---------|
| 実行日時 | Clock抽象化して固定時刻注入。`new Date()` 直書き禁止 |
| 乱数 | seed固定 or 注入 |
| 環境変数 | テスト内で明示的に set/restore |
| 実行順 | 共有state禁止、並列実行で同結果 |
| ファイルシステム | tempdir限定、固定パス禁止 |

## AI テスト生成の原則

### 期待値の導出元

```
期待値 ← 仕様（spec, 要件定義, ドキュメント）  ← 正しい
期待値 ← 実装コードの計算結果                   ← 禁止（カンニング）
```

実装を見て期待値を転写すると、バグを「正解」として固定化する（トートロジカルテスト）。

### コンテキスト分離

| 渡すもの | 渡さないもの |
|---------|------------|
| 仕様書（Given/When/Then） | 実装コード（src/） |
| 型定義（interface, .d.ts） | 計算ロジック |
| テスト設定（jest.config等） | |
| 既存テスト例（パターン統一） | |

### テスト失敗時の判定

```
テスト失敗
├── テストが仕様に合致 → 実装がバグ（実装を修正）
└── テストが仕様に不合致 → テスト修正
```

テストを実装に合わせて修正してはならない。仕様が正。
