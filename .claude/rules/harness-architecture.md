# HARNESS-ARCHITECTURE: Harness 設計アーキテクチャ仕様

各スキル・フックの実装詳細は各 SKILL.md / スクリプトを参照。本ドキュメントは単体では読み取れない設計意図・構成要素間の境界・設計原則の適用基準を記述する。

**責務分担ポインタ**: Harness 全体構造の概要 → CLAUDE.md（Harness Architecture 原則セクション）、開発サイクルのコナセンス → `dev-cycle.md`、設定管理のコナセンス → `config-management.md`、本ドキュメントは上記3文書では読み取れない領域定義・原則適用基準・自己メンテナンスの運用を担う。

**更新タイミング**: 領域定義の境界を変更したとき / 4原則の適用基準を更新したとき / 新カテゴリのスキル・フックを追加したとき / Steering Loop の運用サイクルを変更したとき

## 1. 3領域定義

Harness の構成要素はすべて以下の3領域のいずれかに属する。領域をまたぐ処理は明示的な連携インターフェース（判定キー・ファイル契約）を経由する。

| 領域 | 定義 | 境界条件 |
|------|------|---------|
| **コード生成** | ソースコード・設定ファイル・ドキュメントを生成・変更する処理 | Claude が直接ファイルを Write/Edit するすべての操作 |
| **環境操作** | シェルコマンド・スクリプト・ツール実行を通じて外部状態を変化させる処理 | Bash tool 経由のコマンド実行、git操作、パッケージインストール等 |
| **自己管理** | Harness 自身の構成要素（スキル・フック・ルール）を診断・更新・最適化する処理 | `claude-config`、`skill-creator`、`setup-linter-hooks` 等が担う |

領域の分離原則: コード生成フェーズで環境操作を行わない。環境操作の結果を自己管理の入力とする（劣化検出の起点）。

## 2. 4原則の詳細展開

CLAUDE.md の Harness Architecture 原則宣言を各領域への適用基準として展開する。

### 原則1: Guide First, Sense Second

事前方向づけ（スキルの SKILL.md、rules/、CLAUDE.md）を優先し、事後検出（PostToolUse フック）は逸脱検出に限定する。

| 領域 | 適用基準 |
|------|---------|
| コード生成 | スキルの指示・テンプレートで事前に構造を決定。PostToolUse でのドリフト修正は禁止 |
| 環境操作 | guard スクリプトによる事前チェック（PreToolUse）を優先。事後のロールバックは設計失敗のサイン |
| 自己管理 | 診断より設計ガイド（rules/）の整備を優先。診断は「逸脱が起きた後」の用途に限定 |

### 原則2: Computational First

計測可能・決定論的な基準はスクリプト化する。AI 判断（Inferential）は人間の言語判断が必要な処理のみ。

| 領域 | Computational（スクリプト化） | Inferential（AI 判断） |
|------|-----------------------------|-----------------------|
| コード生成 | ファイル存在確認、行数カウント、diff 抽出 | レビュー観点の優先度付け、設計判断の評価 |
| 環境操作 | lint/test 実行、終了コード判定、パス検証 | エラーメッセージの診断、リカバリ方針の決定 |
| 自己管理 | 行数超過検出、パス参照の整合性確認 | スキル分割タイミングの判断、コナセンス重要度評価 |

### 原則3: ライフサイクル配置

制御をタスクの各フェーズに配置する。単一フックへの集中は単一障害点になる。

| フェーズ | タイミング | 配置する制御の種類 |
|---------|-----------|-----------------|
| タスク開始前 | PreToolUse / スキル冒頭 | 前提条件チェック、禁止操作ガード |
| 実行中 | スキル内ステップ | 品質ゲート通過判定、分岐制御 |
| 実行後 | PostToolUse / スキル末尾 | 成果物検証、次フェーズへの引き継ぎ |
| セッション後 | Stop hook | セッションサマリ記録、一時ファイルクリーンアップ |
| 定期 | Cron / 手動トリガー | Harness 劣化検出、依存関係の陳腐化確認 |

### 原則4: 自己メンテナンス

Harness 自体が劣化する。設計時点で劣化検出と修正の仕組みを組み込む。詳細は「5. Steering Loop」を参照。

## 4. 責務マップ

スキル・フックをカテゴリ × 領域のマトリクスで分類する。具体的なスキル名・フック名は記載しない（陳腐化対策）。追加・削除に対してこのマップは更新不要。

| カテゴリ | コード生成 | 環境操作 | 自己管理 |
|---------|-----------|---------|---------|
| **計画・設計** | 設計書・タスクリストの生成 | - | - |
| **実装・変換** | ソースコード・設定ファイルの生成 | コマンド実行・ファイル移動 | - |
| **検証・品質** | レビュー結果のドキュメント化 | lint・test・build の実行 | - |
| **診断・修復** | バグ報告・修正案の生成 | デバッグコマンドの実行 | Harness 設定の診断・修正 |
| **セッション制御** | - | フック経由のガード・記録 | スキル選択・コンテキスト管理 |

カテゴリが複数領域にまたがる場合、主たる責務で分類する。複数領域の連携は領域間インターフェース（判定キー・ファイル契約）で記述する。

## 5. Steering Loop

自己メンテナンス原則の運用サイクル。Harness 自体の劣化（設定の陳腐化・スキル肥大化・コナセンス破綻）を継続的に検出・修正する。

```mermaid
flowchart LR
    D[劣化検出] --> N[診断]
    N --> F[修正]
    F --> V[検証]
    V --> D
```

### 知見フロー

3つの知見源から蓄積→昇格の2段構えで rule/skill/CLAUDE.md に還元する。

```
知見源              捕捉(自動化)      蓄積          昇格(人間レビュー)

retrospective ───→ memory ─────┐
逸脱FB ──────────→ memory ────┤──→ claude-config ──→ rule/skill/CLAUDE.md
opusレビュー ────→ tmp/review ─┘    (昇華+頻出抽出)    (価値観一致確認)
```

**責務分離の原則**:
- **retrospective**: 知見の「捕捉」に専念（memory 蓄積のみ）。計測データ（metrics_summary）は事実の記録に留め、評価・昇格は行わない
- **claude-config**: 「昇格」に専念（memory/tmp → rule/skill/CLAUDE.md）。scan-review.py で tmp/review/ の頻出パターンを抽出し、昇格候補として診断レポートに含める

### 計測データの流れ

| データ | 収集 | 蓄積先 | 消費 |
|--------|------|--------|------|
| PermissionRequest | hook（log-permission-request.sh） | `~/.claude/tmp/metrics/permission-requests.jsonl` | collect.py → metrics_summary → retrospective サブエージェント |
| opusレビュー結果 | implement / claude-config のレビューサイクル | `~/.claude/tmp/review/*.md` | scan-review.py → claude-config 診断 |

### 劣化のシグナル

| シグナル | 検出方法 | 対応カテゴリ |
|---------|---------|------------|
| スキルが期待どおりに起動しない | description のトリガー漏れを確認 | スキル設計の見直し |
| フックが無効化されたまま残存 | settings.json とスクリプト実体の整合確認 | 設定管理のコナセンス |
| rules/ ファイルが肥大化 | 行数 200 超のファイルを検出 | skills/ への分離 |
| 同一情報の重複 | references/ と rules/ の並走を確認 | 正規ソースへの集約 |
| PermissionRequest の偏り | metrics_summary の tool_counts 集計 | permissions.allow / guard の見直し |
| opusレビュー指摘の頻出 | scan-review.py の frequent_dimensions | rule/skill への昇格検討 |

### 修正の責務分担

- **診断**: `claude-config` スキル（自己管理領域）が担う
- **スキル設計の修正**: skill-design-patterns の設計パターン（claude-config/references/skill-design-patterns.md）に基づき対応。スキル作成の実行は skill-creator が担う
- **フック・settings.json の修正**: `config-management.md` のコナセンス・マップに従う
- **本ドキュメント自体の修正**: Harness 設計の境界・原則が変わったときのみ実施
