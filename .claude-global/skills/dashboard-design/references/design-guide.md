## 1. 3層情報階層の設計基準

### レイヤー定義

| Layer | 役割 | メトリクス数 | 表示条件 |
|-------|------|------------|---------|
| **Layer 1: Overview** | 3–5秒で全体像把握 | 5–7個 | 常時表示 |
| **Layer 2: Detail** | 重要だが低頻度のデータ | 制限なし | 展開時のみ（Progressive Disclosure） |
| **Layer 3: Action** | 根本原因分析 → 次ステップ提示 | — | ドリルダウン後 |

### 判断基準

**Layer 1 に入れるかどうかの問い**:
- 「この数値が毎日変わっても、誰かが毎日確認する必要があるか？」→ Yes なら Layer 1 候補
- Decision Test（SKILL.md §1）で3問すべてに答えられるか → No なら Layer 2 以下へ

**レイアウト原則（F/Z-パターン）**:
- 左上: 最重要KPI（目が最初に向かう位置）
- 右下: 補助的な文脈情報

**Layer 3 の設計原則**:
- 根本原因分析はダッシュボード内で完結させる（別ページ遷移は離脱を招く）
- 「なぜ」に答えた後、次のアクションを提示する（「確認」で終わらせない）

## 2. Leading/Lagging 分離の配置基準

### 配置マトリクス

| 層 | 更新頻度 | 指標種別 | 代表的な指標例 | 所有者 |
|----|---------|---------|--------------|--------|
| **戦略層** | 月次〜四半期 | Lagging | MRR, Churn, NPS, LTV | Executive / CPO |
| **実行層** | 週次 | Leading | Feature Adoption, Task Success | Product Manager / Squad Lead |
| **運用層** | 日次 | 直接制御可能 | API Latency, Error Rate | Engineer / Operator |

### 配置の判断フロー

```
指標を配置しようとしている → その指標は何を予測する？
  ↓
「将来の状態を予測する」→ Leading → 実行層
「過去の結果を確認する」→ Lagging → 戦略層
「今すぐ操作で変えられる」→ 直接制御可能 → 運用層
```

**Gotcha**: Lagging 指標を運用層に置いても誰も行動できない（制御手段がない）。配置層と所有者の権限が一致していない場合、そのメトリクスは「見るだけ」になる。

### Leading → Lagging 予測チェーン

Leading 指標が Lagging 指標を予測できる構造を設計時に明示する。

```
Leading:  Feature Adoption Rate
  ↓ 1–2週間後に影響
Lagging:  Retention Rate
Trigger:  Feature Adoption < 20% → オンボーディング改善着手
```

この構造がないと、Lagging の悪化を見てから「なぜ？」を遡る後追い分析になる。

## 3. 文脈付与パターン（比較軸の選択基準）

各メトリクスに最低1つの比較軸を必須とする。比較軸なしの数値は「良いのか悪いのか」の判断基準を与えない。

### 比較軸の選択基準

| 比較軸 | 表示例 | 使うべき状況 | 使わない状況 |
|--------|--------|------------|------------|
| **目標値** | 達成率 75%（目標: 80%） | 目標が明確に設定されている | 目標値がまだ決まっていない |
| **前期間比** | 前週比 +23% | 変化の方向・速度を見たい | 季節性が強くて前週比が無意味な場合 |
| **トレンド（スパークライン）** | 折れ線ミニグラフ | 短期変動 vs 長期傾向を区別したい | 単発イベントの数値（累積不適） |
| **異常表示** | ±2σ 逸脱時ハイライト | 閾値を事前に決めたくない動的指標 | 月次など低頻度でサンプル数が少ない指標 |

### 優先順位の判断基準

1. **目標値が存在する** → 目標値を第一比較軸にする（達成/未達が即判断できる）
2. **目標値がない、週次以上** → 前期間比を使う（変化の方向を見る）
3. **変動が激しい日次以上** → トレンド（スパークライン）を併用する
4. **成長中で閾値設定が難しい** → 異常表示（動的閾値 ±2σ）を使う

**複数軸の組み合わせ例**:
```
Conversion Rate: 3.2%（目標: 4.0%）前週比 -0.3pt [スパークライン]
```
目標値 + 前期間比 + トレンドの3軸は「達成率・変化・傾向」をワンラインで提供する。

## 4. メトリクス→アクション導線の事前定義

ダッシュボードに載せる**前に**、各メトリクスの行動導線を設計する。設計後に「このメトリクスが動いたら誰が何をするか？」に答えられない場合は搭載しない。

### パターン A: Signal → Insight → Decision → Action

```
Signal:   広告支出ペーシングが予算の 130%
Insight:  自動入札が低品質プレイスメントで過剰支出
Decision: 入札削減、予算再配分
Action:   入札ルールを更新する
```

このパターンは「数値を見て行動するまでの思考チェーン」を事前に定義する。Signal で止まっているダッシュボードは観察ツールにすぎない。

### パターン B: Leading → Lagging 予測チェーン

「§2. Leading/Lagging」の配置基準と組み合わせる。Leading が閾値を下回った時点でアクションをトリガーする。

```
Leading:  Feature Adoption Rate 35%
  ↓ 1–2週間後に影響
Lagging:  Retention Rate
Trigger:  Adoption < 20% → オンボーディング改善着手（待たずに先行対応）
```

### パターン C: 閾値 → エスカレーション

```
Warning:  Error Rate > 5%  → チームへの通知
Critical: Error Rate > 10% → オンコール担当へエスカレーション
```

**段階的エスカレーションの原則**: 対応できる権限を持つ人だけに通知する。全員通知はアラート疲弊の主因。

## 5. 認知負荷低減テクニック

### テクニック一覧

| テクニック | 実装方法 | 根拠・効果 |
|-----------|---------|-----------|
| **Progressive Disclosure** | ホバー時ツールチップ、展開パネル | 初期表示の情報量を削減しつつ詳細へのアクセスは保持 |
| **Data-Ink Ratio 最大化** | 装飾排除、3D禁止、不要枠線削除 | 情報と無関係のインクを排除し、読み取りコストを下げる |
| **3秒ルール** | 最重要メトリクスを超大型で表示 | 技術レベル不問で「今どうなっているか」を即把握可能にする |
| **一貫レイアウト** | タイトル左上・凡例下中央を固定 | 毎回のレイアウト学習コストをゼロにする |
| **意味のある色のみ** | 装飾目的の色を禁止、状態表現にのみ使用 | 色弱対応 + 情報と装飾の混同を防ぐ |
| **Hick's法則の適用** | フィルタを段階開示（必須のみ最初表示） | 選択肢の増加が意思決定を遅らせる問題を防ぐ |

### 判断基準: 「情報追加」 vs 「情報削除」

追加を検討するとき:
- 「このビジュアル要素を削除すると理解が損なわれるか？」→ No なら削除

削除を検討するとき:
- Layer 1 が7個を超えた → 最も使用頻度が低いものを Layer 2 へ移動
- 色が3種類以上 → 「状態（OK/Warning/Critical）」の表現以外は除去

## 6. Gotchas

| 状況 | 落とし穴 | 対処 |
|------|---------|------|
| 新規ダッシュボード設計時 | 「あれもこれも」でメトリクスが増殖する | 初期は3–5個に絞る。追加はDecision Testを通過したもののみ |
| Lagging 指標の悪化を検知 | 原因が追えず「様子見」になる | Leading 指標との予測チェーンを事前に設計しておく（§2） |
| Leading 指標だけ追う | 戦略的な成果との接続が見えなくなる | 戦略層のLagging指標と定期的に突き合わせる |
| 比較軸なしの数値 | 「増えた/減った」しか言えない | 必ず1軸以上の比較を付ける（§3の選択基準を参照） |
| アラートを全員に送る | アラート疲弊 → 無視が常態化 | 対応権限のある人だけに段階的に送る（§4 パターンC） |
| フィルタ・ドリルダウンを増やす | ダッシュボードが分析ツール化する | ダッシュボードと分析ツールは用途を分離する |
| 設計後に放置する | データが古くなり信頼を失う | 四半期レビュー（ASSESS→PLAN→ACT→REVIEW）をライフサイクルに組み込む |
