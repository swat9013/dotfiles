## 1. Stage別詳細採点基準

SKILL.md の4段階診断フロー各Stageで用いる判定基準。

### Stage 1: 現状把握

**メトリクス分類シート**

| 軸 | 分類方法 | 判定境界 |
|---|---------|---------|
| Leading/Lagging | 「このメトリクスが変化してから結果が出るまでどれくらいかかるか」を問う。即日〜数日 = Leading、数週〜数ヶ月 = Lagging | 境界が曖昧な場合は両方マークして Stage 2 で判断 |
| レイヤー | 更新頻度と所有者で判定。日次+Engineer = 運用層、週次+PM = 実行層、月次以上+Executive = 戦略層 | 所有者不明はレイヤー分類不可 → Stage 2 で「存在理由不明」候補 |
| 表示順序 | 左上/最上部 = 最重要扱い。その配置が実際の重要度と一致するか確認 | 重要度と配置が逆転している = Vanity 優先配置アンチパターン |

### Stage 2: 目的確認（Decision Test 逆適用）

設計時は「搭載可否判定」だが、診断時は「存在理由の逆証明」として使う。

| 問 | 逆適用の問い | OK | 改善余地あり | 要対応 |
|---|------------|-----|------------|-------|
| Q1: 誰が何をする？ | 「このメトリクスが閾値を超えた/下回ったとき、具体的に誰が何をするか言えるか」 | 担当者名+アクション内容が即答できる | 「チームが確認する」など曖昧 | 誰も答えられない |
| Q2: OKR/KPI寄与 | 「このメトリクスが改善すると、どのOKR/KPIの数字が動くか」 | 上位指標への寄与が明確 | 間接的に関係するが経路が曖昧 | 関係が説明できない |
| Q3: 所有者 | 「このメトリクスが悪化し続けた場合、誰が責任を持って対処するか」 | 固有の担当者名が即答できる | チーム名は言えるが個人不明 | 誰も所有していない |

**判定ルール**: 3問中1問でも「要対応」= メトリクス全体を「要対応」にマーク。3問すべて「改善余地あり」以上 = 「改善余地あり」。

### Stage 3: 3A評価

各基準の具体的な問いと判定境界。

**Actionable**

| 判定 | 具体的な問い | 境界 |
|-----|------------|-----|
| OK | 「X を Y に変えたら、このメトリクスは Z 方向に動くと予測できるか」 | 因果関係を1文で説明できる |
| 改善余地あり | 相関は見えるが介入手段が不明確 | 「影響しそう」止まり |
| 要対応 | アクションと結果の連鎖が全く説明できない | 観察するだけで終わる |

**Accessible**

| 判定 | 具体的な問い | 境界 |
|-----|------------|-----|
| OK | 「このダッシュボードを初めて見る非技術者が、この数値が良いのか悪いのかを3秒で判断できるか」 | 単位・方向・目標値が揃っている |
| 改善余地あり | 文脈があれば理解可能だが初見では難しい | 説明が必要 |
| 要対応 | ドメイン専門知識なしでは解釈不能 | 用語定義が必要 |

**Auditable**

| 判定 | 具体的な問い | 境界 |
|-----|------------|-----|
| OK | 「この数値の計算ロジック・データソース・更新タイミングを今すぐ確認できるか」 | クエリ/定義へのリンクが存在する |
| 改善余地あり | 誰かに聞けばわかるが文書化されていない | 口頭伝承の状態 |
| 要対応 | 計算ロジックが不明・再現不能 | 「たぶんこう計算している」レベル |

### Stage 4: 改善アクション選定基準

最大3件に絞る際の優先度付け基準。

| 優先度 | 条件 | 根拠 |
|-------|-----|------|
| 高 | 「要対応」かつ Q1（誰が何をする）が答えられない | 行動導線の完全欠如は最大リスク |
| 中 | Vanity metric が Overview の上位に配置されている | 閲覧者の意思決定を誤誘導し続けている |
| 低 | Accessible は OK だが比較軸が1つしかない | 判断可能だが精度が低い |

## 2. Signal→Action導線評価

メトリクスが実際の行動に接続しているかを確認するフロー。

### 導線パターン別評価基準

**パターンA: Signal→Insight→Decision→Action**

```
Signal:   広告支出ペーシングが予算の130%
Insight:  自動入札が低品質プレイスメントで過剰支出
Decision: 入札削減、予算再配分
Action:   CMS側で入札ルール更新
```

| 評価点 | 問い |
|-------|-----|
| Signal明確性 | Signal の数値変化が即座に判断可能か（目標値・前期間比・閾値が揃っているか） |
| Insight到達性 | Signal だけで Insight まで辿り着けるか、追加ドリルダウンが必要か |
| Decision権限 | Decision を下す人物がこのダッシュボードを見ているか |
| Action完結性 | Action をダッシュボード内から実行/起票できるか、外部ツールへのリンクがあるか |

**パターンB: Leading→Lagging予測チェーン**

```
Leading:  Feature Adoption Rate 35%
  ↓ (1-2週間後の予測)
Lagging:  Retention Rate
Trigger:  採用率 <20% → オンボーディング改善着手
```

| 評価点 | 問い |
|-------|-----|
| 予測チェーン定義 | Leading と Lagging の因果関係が明示されているか（コメント・ツールチップ等） |
| Trigger設定 | 行動起点の閾値が事前定義されているか |
| 予測ラグ表示 | Leading から Lagging への時間差が表示されているか |

**パターンC: 閾値→エスカレーション**

```
Warning:  Error Rate > 5%  → チームチャンネル通知
Critical: Error Rate > 10% → on-call 呼び出し
```

| 評価点 | 問い |
|-------|-----|
| 閾値の根拠 | 静的閾値か動的閾値か。静的なら成長・季節変動で陳腐化していないか |
| エスカレーション先 | 通知先が現在も有効か（担当者変更・チャンネル廃止等） |
| False Positive率 | 過去30日の真陽性率を把握しているか |

### 導線なし（「見るだけで終わる」）の特定

以下がすべて当てはまる場合、導線なしと判定する:

- Q1（誰が何をする）に誰も答えられない
- 閾値・アラート・通知が設定されていない
- ドリルダウンしても次のアクションへのリンクがない
- 最後にアクションが取られた記録がない（利用ログ確認）

**判定後の対処**: 即削除ではなく「30日間誰も参照しなければ削除」という猶予ルールを推奨。突然の削除は政治的摩擦を生む。

## 3. 認知負荷チェック

### Progressive Disclosure 適用判定

| 適用すべき場合 | 適用不要な場合 |
|-------------|-------------|
| 詳細データが存在するが毎回必要ではない | すべての閲覧者が毎回詳細を必要とする |
| Overview で OK/NG の判断が可能 | 詳細なしでは判断不能なメトリクス |
| 展開前後でメトリクス数が3倍以上変わる | 展開コストが認知節約効果を上回る |

### Data-Ink Ratio チェックリスト

削除して情報量が変わらないものは削除する:

- [ ] 3D グラフ（奥行きで値が歪む）
- [ ] 装飾的な背景色・グラデーション
- [ ] 軸ラベルと凡例の重複（どちらか一方で足りる）
- [ ] ゼロからでない Y 軸（変化を誇張/矮小化する）
- [ ] 小数点以下 3 桁以上（精度が意味を持たないケース）
- [ ] 枠線のみの役割をするグリッド線（色を薄くするか削除）

### 3秒ルール確認

最重要メトリクスが即座に視認できるかを確認する:

1. ダッシュボードを3秒だけ表示して閉じる
2. 「今日この組織は良い状態か？」に答えられるか
3. 答えられない場合 → 最重要情報の視覚的階層が崩れている

**修正基準**: 最重要メトリクスのフォントサイズ・配置・色が他の要素より明確に優先されているか。

### 色の使用基準

| 用途 | OK | NG |
|-----|-----|-----|
| 状態表現 | 赤=警告/超過、緑=正常、黄=注意（意味に紐づく） | 装飾目的の配色、「見栄えをよくする」色 |
| カテゴリ区別 | 意味のある分類を色で示す | 似た色で微妙な違いを表現 |
| 強調 | 最重要1〜2点のみ強調色を使う | 5点以上が同じ強調色 |

**色弱対応**: 赤/緑のみで状態を区別しない。形状・ラベル・パターンを補助的に使う。

### Hick's法則によるフィルタ設計

| フィルタ数 | 推奨アプローチ |
|----------|-------------|
| 1〜3 個 | 全表示でよい |
| 4〜7 個 | 「よく使う」と「詳細設定」に分割して段階開示 |
| 8 個以上 | フィルタ設計の見直しを推奨。カテゴリ階層化またはプリセット化 |
