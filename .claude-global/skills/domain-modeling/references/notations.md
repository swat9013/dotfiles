## UML クラス図

業界標準の概念モデル表現。エンティティ・値オブジェクト・関係性の明確化に優れる。

### ドメインモデリングでの活用

- Entity: ライフサイクルを持つオブジェクト（ID で識別）
- Value Object: 属性の組み合わせで同一性を判断（不変）
- Aggregate Root: 外部からのアクセスポイント
- 関連・集約・コンポジション関係の表現

### 利点

- 構造と関係の厳密な表現が可能
- 広く理解されている標準記法
- 多数のツールがサポート

### 欠点

- 大規模モデルでは図が複雑化
- 振る舞い（ビジネスルール）の表現が弱い
- 非技術者には読みにくい

## C4 モデル

Simon Brown 提唱の4段階アーキテクチャ表現。抽象度に応じた図を段階的に提供する。

### 4レベル

| レベル | 対象 | 対象読者 |
|--------|------|----------|
| System Context | システムと外部アクター・システムの関係 | 全ステークホルダー |
| Container | アプリ、DB、メッセージキュー等の技術的構成要素 | 技術リーダー、アーキテクト |
| Component | コンテナ内のコンポーネント | 開発者 |
| Code | クラス・インターフェースレベル（通常は自動生成） | 開発者 |

### ドメインモデリングとの関係

- System Context ≒ DDD の Context Map（外部システムとの関係）
- Container ≒ Bounded Context の技術的実現
- Component ≒ Aggregate / Module の構成

### 利点

- 技術者・非技術者の共通理解形成に効果的
- 段階的な詳細化が可能
- ツール非依存（テキストでもホワイトボードでも）

### 欠点

- ドメインロジックの表現には不向き
- Code レベルは実用性が低い（コードを直接読むほうが早い）
- ドメインモデルとの直接的な対応が弱い

## Context Map

DDD の Bounded Context 間の関係を図式化する記法。

### 9つの関係パターン

| パターン | 記号 | 意味 |
|---------|------|------|
| Shared Kernel | SK | 共有モデル（密結合チーム間） |
| Customer/Supplier | C/S | 上流/下流の依存関係 |
| Anti-Corruption Layer | ACL | 変換層で外部モデルを隔離 |
| Open Host Service | OHS | 公開プロトコルでサービス提供 |
| Published Language | PL | 標準化された交換形式 |
| Conformist | CF | 上流モデルに従属 |
| Separate Ways | SW | 統合しない |
| Partnership | PS | 協調的な開発 |
| Big Ball of Mud | BBoM | 境界不明確（回避対象） |

### ツール

- [Context Mapper](https://contextmapper.org/) — DSLベースの Context Map 作成ツール

## Mermaid

Markdown内に記述できる軽量ダイアグラム記法。GitHubが直接レンダリングをサポート。

### ドメインモデリングでの活用

- クラス図: Entity/VO/Aggregate の構造表現
- シーケンス図: ドメインイベントのフロー
- フローチャート: ビジネスプロセス

### 利点

- Git管理可能（テキストベース）
- GitHub/GitLab で直接レンダリング
- 学習コストが低い
- Markdown ドキュメントに直接埋め込み可能

### 欠点

- 複雑な図の表現力に限界
- レイアウト制御が限定的
- UML仕様の完全準拠ではない

---

## PlantUML

テキストベースの UML 作成ツール。UML仕様への準拠性が高い。

### 利点

- UML仕様に厳密に準拠
- Git管理可能
- 長期安定性（10年以上の実績）
- 規制対応（厳密な仕様書作成）向け

### 欠点

- Java ランタイムが必要
- レンダリングに外部ツール/サーバーが必要
- Mermaid より学習コストが高い

---

## draw.io

完全無料のGUIベースダイアグラムツール。VS Code拡張も利用可能。

### 利点

- GUI直感的操作
- 多様な図種に対応（UML、ER図、フローチャート等）
- XML形式で Git 管理可能（差分は読みにくい）
- VS Code 拡張でエディタ内編集

### 欠点

- テキストベースの差分レビューが困難
- 自動レイアウトの品質にばらつき
- 大規模チームでの同時編集に制約

## ADR

Architecture Decision Records。設計判断の理由と代替案を永続化するドキュメント形式。

### 構成

```
# ADR-NNN: タイトル

## ステータス
提案 / 承認 / 非推奨 / 置換

## コンテキスト
判断が必要になった背景

## 決定
採用した選択肢と理由

## 代替案
検討したが採用しなかった選択肢と却下理由

## 結果
この決定による影響・トレードオフ
```

### 利点

- 「なぜそう設計したか」が永続化される
- 新メンバーのオンボーディングに有効
- Git管理可能

### 欠点

- 記述の継続が組織的な習慣化を要する
- 過去のADRが陳腐化するリスク
- ドメインモデル自体の表現手段ではない（補助ツール）
